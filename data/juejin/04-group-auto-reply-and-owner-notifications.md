# 让 QQ AI 机器人“可控地插话”：群聊主动回复与主人转告设计

> 本文基于项目 v0.7 和 v0.8 的实现记录整理，重点介绍 QQ AI 机器人如何在授权群里进行可控主动回复，以及如何用固定命令把短文本转告给主人。这里的主动能力不是让 AI 自由监听和随机发言，而是在明确权限、评分、冷却和隐私边界下，小步扩展机器人的运行能力。

## 摘要 / 导语

当 QQ AI 机器人只在私聊或被 @ 时回复，系统相对简单：用户发起，机器人响应。但一旦想让机器人在群里“主动插一句”，复杂度会立刻上升。它可能刷屏、打扰群聊、误读上下文、泄露主人隐私，甚至变成一个骚扰入口。

因此 v0.7 和 v0.8 没有直接做“AI 自由主动聊天”，而是拆成两个更可控的能力：

```text
v0.7：群聊主动回复，只在明显需要回应时触发
v0.8：主人通知与转告，只接受明确命令和短文本
```

这篇文章介绍这两个模块的设计：如何判断群聊里是否该回复，如何避免频繁插话，如何限制转告内容，以及为什么这些主动能力都不应该绕过权限和安全规则。

## 主动能力为什么要克制

AI 机器人一旦进入群聊，会比私聊复杂得多。

私聊里通常只有一个用户和机器人，触发关系清楚。群聊里则有很多噪声：

- 有人在闲聊。
- 有人在开玩笑。
- 有人在提到机器人名字但不一定想让它回复。
- 有人可能故意诱导机器人泄露信息。
- 群里可能同时出现主人、白名单用户、普通群友和黑名单用户。

如果简单写成：

```text
群里每条消息都交给 AI 判断要不要回复
```

会有几个问题：

- 调用成本变高。
- 响应延迟增加。
- 群聊容易被刷屏。
- 大模型可能误判玩笑、反话和上下文。
- 安全边界变得不可控。

所以第一版主动能力必须克制。

项目里采用的原则是：

```text
本地规则先判断是否值得回复。
达到阈值后，才调用正常聊天模型。
```

也就是说，主动回复模块只负责“是否触发”，不负责“怎么回答”。

## v0.7：群聊主动回复

v0.7 的目标是在授权群内提供非 @ 回复能力。

它不是让机器人监听所有群聊然后自由发言，而是只在白名单群、明确命中规则、冷却允许的情况下插话。

第一版支持：

```text
白名单群内非 @ 消息主动回复
本地规则评分
群全局冷却
用户冷却
主人冷却
角色卡专属别名和触发词
```

第一版明确不做：

```text
非授权群主动回复
随机插话
主动开话题
AI 自主判断是否回复
读取图片/语音/文件触发主动回复
绕过权限、长度限制和黑名单
```

这个范围控制很重要。

主动回复如果一开始就设计成“AI 自由判断”，后续很难排查为什么它突然插话。用规则评分虽然不完美，但可解释、可调参、可回滚。

## 配置开关

主动回复默认关闭：

```env
ENABLE_GROUP_AUTO_REPLY=false
GROUP_AUTO_REPLY_THRESHOLD=50
GROUP_AUTO_REPLY_COOLDOWN_SECONDS=60
GROUP_AUTO_REPLY_OWNER_COOLDOWN_SECONDS=30
GROUP_AUTO_REPLY_USER_COOLDOWN_SECONDS=120
```

几个配置的含义：

- `ENABLE_GROUP_AUTO_REPLY`：是否启用非 @ 主动回复。
- `GROUP_AUTO_REPLY_THRESHOLD`：评分达到多少才回复。
- `GROUP_AUTO_REPLY_COOLDOWN_SECONDS`：同一群的全局冷却。
- `GROUP_AUTO_REPLY_OWNER_COOLDOWN_SECONDS`：主人触发主动回复的冷却。
- `GROUP_AUTO_REPLY_USER_COOLDOWN_SECONDS`：普通用户触发主动回复的冷却。

默认关闭是为了避免功能上线后立刻影响真实群聊。

更稳的流程是：

```text
先在测试群开启
观察触发频率
调整阈值和冷却
确认不刷屏后再扩大范围
```

## 基础过滤：先决定哪些消息绝不处理

进入评分前，消息要先过一层基础过滤。

不会触发主动回复的情况包括：

```text
机器人已被 @
非白名单群
命令消息
空文本
超出群聊长度限制
黑名单用户
群聊功能未开启
主动回复开关关闭
```

这里有一个细节：如果机器人已经被 @，就不走主动回复，而是走普通群聊 @ 回复逻辑。

这样可以避免同一条消息被两套逻辑重复处理。

## 规则评分：为什么不用 AI 判断

第一版主动回复不额外调用大模型判断“该不该回复”。

原因很实际：

- 省成本。
- 响应更快。
- 结果可解释。
- 误触发时能直接调规则。
- 不增加一层提示词安全问题。

本地评分主要看这些加分项：

```text
提到机器人名称或角色卡别名
出现召唤词，比如“出来”“在吗”“看看”
消息内容是问题
主人发言
主人在群里提问
主人自我否定
有人诋毁主人
消息明显是在求助
```

也会通过短句、闲聊噪声、不明确对象等情况降低触发概率。

最终逻辑是：

```text
score >= GROUP_AUTO_REPLY_THRESHOLD
```

才允许进入正常聊天流程。

## 角色卡专属触发词

不同角色卡可能有不同称呼和别名。

因此主动回复配置不应该硬编码在代码里，而是放在角色卡配套 JSON 中：

```text
prompts/persona-cards/<role-key>.auto-reply.json
```

它可以包含：

```json
{
  "bot_aliases": ["机器人"],
  "call_markers": ["在吗", "出来", "看看"],
  "question_markers": ["吗", "？", "?"],
  "help_markers": ["怎么", "为什么", "帮我"],
  "owner_target_markers": [],
  "insult_markers": [],
  "self_negative_markers": []
}
```

这样切换角色卡时，主动回复触发词也可以跟着变化。

它还有一个额外好处：真实角色卡和真实触发词可能包含隐私设计，可以放在本地私密目录，不必提交到公开仓库。

## 冷却策略：防止刷屏

主动回复必须有冷却。

项目里用了三层冷却：

```text
群全局冷却
普通用户冷却
主人专属冷却
```

普通用户触发后，会受用户冷却限制。群内任意主动回复也会受群全局冷却限制。

主人触发时可以更灵敏一些，但仍然有主人专属冷却，避免机器人在主人连续发言时频繁插话。

这套策略的重点是：

```text
主人更优先，但不是无限优先。
```

群聊机器人最怕“热心过头”。冷却比复杂判断更能直接控制体验。

## 主动回复和普通聊天的关系

主动回复只决定“要不要回复”。

一旦决定回复，后续流程和普通群聊一致：

```text
构建会话上下文
注入当前发言者身份
注入主人公开信息规则
加载基础提示词和角色卡
读取会话摘要和手动长期记忆
调用聊天模型
保存用户消息和机器人回复
触发自动摘要压缩
```

这点非常关键。

主动回复不能成为新权限层，也不能绕过安全规则。

它只是触发层。

## v0.8：主人通知与转告

v0.8 做的是另一个“主动”能力：

```text
允许授权用户用明确命令向主人发送短文本转告。
```

命令是：

```text
/转告主人 内容
/留言给主人 内容
```

两个命令行为一致。

例如：

```text
/转告主人 服务器异常，请看群。
/留言给主人 明天十点开会。
```

机器人收到后，会主动给主人发私聊通知。

但这里也有严格边界：这不是 AI 自主判断是否找主人聊天，而是用户明确发出命令。

## 为什么转告不走 LLM

转告最重要的是准确和可追溯。

如果让 AI 改写或摘要，就可能出现这些问题：

- 把原意改轻或改重。
- 漏掉关键信息。
- 添加原文没有的情绪。
- 让角色卡影响转告语气。
- 在敏感内容里做错误判断。

所以 v0.8 的第一版设计是：

```text
转告不调用 LLM
不走角色卡
不润色
不摘要
不自动截断
```

它只做三件事：

```text
检查权限
检查内容
按固定模板发给主人
```

## 转告权限

允许触发转告的来源：

```text
主人本人
私聊白名单用户
授权群里的群成员
```

禁止触发转告的来源：

```text
黑名单用户
非授权群
陌生私聊用户
```

也就是说，转告命令不会变成陌生人骚扰主人的入口。

它仍然继承整个机器人的权限体系。

## 内容限制

转告内容必须满足：

```text
不能为空
必须是纯文本
最多 50 字
不包含明显敏感信息
```

超过 50 字时，固定回复：

```text
请主动联系主人，文本过长不予转告。
```

这里不做自动截断。

原因是截断可能改变语义。比如：

```text
不要重启服务器，除非我确认
```

如果被截断成：

```text
重启服务器
```

就很危险。

同理，也不让 AI 帮忙摘要，因为摘要可能遗漏边界条件。

## 敏感信息拦截

即使内容不超过 50 字，也要拦截明显敏感内容。

第一版拦截：

```text
密码
验证码
Token
API Key
身份证号
手机号
二维码
数据库内容
```

命中时回复：

```text
内容包含敏感信息，不予转告。
```

这里的边界是：

```text
危险请求可以转告
敏感数据本身不转告
```

例如：

```text
/转告主人 有人想让主人发 token
```

可以转告。

但：

```text
/转告主人 token 是 [密钥内容]
```

不应该转告。

## 固定转告格式

群聊来源：

```text
收到一条转告：
来源：群 群号
发送人：QQ号
内容：转告内容
```

私聊来源：

```text
收到一条转告：
来源：私聊用户 QQ号
内容：转告内容
```

原会话成功提示：

```text
已转告主人。
```

失败提示：

```text
转告发送失败，请稍后再试。
```

不要暴露主人是否在线，也不要把内部异常细节发给普通用户。

## 转告冷却

转告比普通聊天更需要冷却，因为它会主动打扰主人。

第一版冷却：

```env
OWNER_NOTIFICATION_GLOBAL_COOLDOWN_SECONDS=60
OWNER_NOTIFICATION_GROUP_COOLDOWN_SECONDS=120
OWNER_NOTIFICATION_USER_COOLDOWN_SECONDS=300
```

含义：

- 全局给主人 60 秒内最多一次。
- 同一群 120 秒内最多一次。
- 同一用户 300 秒内最多一次。

冷却期间回复：

```text
转告过于频繁，请稍后再试。
```

这可以防止同一个群或同一个用户连续刷给主人。

## 转告和其他模块的边界

### 不走角色卡

转告不应该被角色卡影响。

因为角色卡可能加入称谓、情绪或风格化表达，而转告需要保真。

### 不写长期记忆

转告是一次性消息，不应该自动写入长期记忆。

除非主人明确认为某条信息值得长期保存，再通过长期记忆命令手动添加。

### 不自动转回群聊

主人收到私聊通知后，第一版不自动把主人回复转回原群。

原因是这会引入新的复杂度：

- 如何确认主人想公开回复？
- 如何处理多个群同时转告？
- 如何避免误发隐私？

这些都应该单独设计，不应该混在第一版里。

## 实现结构

转告模块可以单独放在：

```text
src/plugins/ai_chat/owner_notify.py
```

核心函数包括：

```text
validate_owner_notification_content()
contains_sensitive_notification_content()
format_owner_notification()
```

主插件里只负责注册命令：

```python
tell_owner_cmd = on_command(
    "转告主人",
    aliases={"留言给主人", "tell_owner"},
    priority=5,
    block=True,
)
```

这样转告校验、格式化和命令处理分开，后续如果要加转告记录表，也比较容易扩展。

## 后续可以做什么

v0.7 和 v0.8 都是第一版，因此保留了很多后续空间。

主动回复后续可以考虑：

```text
每小时回复上限
更细的群聊白名单策略
按群配置不同阈值
主动回复命中日志
更可解释的评分状态命令
```

转告后续可以考虑：

```text
转告记录表
/查看转告记录
系统错误主动通知主人
白名单申请通知主人
摘要压缩失败通知主人
```

但不建议马上做：

```text
AI 随机找主人聊天
AI 自主判断哪些群聊要告诉主人
主人回复自动转回群聊
自然语言自动识别转告
```

这些能力都涉及更高的隐私和误触发风险，应该等基础权限、日志和审计能力更稳定后再考虑。

## 小结

v0.7 和 v0.8 的共同点是：它们都在做“主动能力”，但都没有把主动权完全交给 AI。

群聊主动回复的边界是：

```text
本地规则评分决定是否触发
正常聊天链路负责生成回复
权限、冷却、安全和角色卡全部照常生效
```

主人转告的边界是：

```text
必须明确命令触发
只允许短文本
不调用 LLM
不走角色卡
不写长期记忆
按固定格式给主人
```

这两个模块看起来不复杂，但它们解决的是机器人从“被动回复”走向“可控主动”的关键一步。

对群聊 AI 机器人来说，主动能力最重要的不是“看起来聪明”，而是：

```text
知道什么时候不该说话
知道什么信息不能转发
知道自己不能绕过权限
知道打扰主人需要节制
```

这也是本地 QQ AI 机器人能长期稳定运行的基础。
