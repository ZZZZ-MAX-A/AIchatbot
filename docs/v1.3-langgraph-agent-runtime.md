# v1.3 LangGraph Agent Runtime 设计

本文记录 v1.3 的目标、边界和设计方案；实际落地状态、QQ live 验证和补丁记录统一整理到 `docs/version-runlog.md`。

本文记录下一版本的 Agent 化设计边界。v1.3 的目标不是立即增加更多功能，而是先规范系统运行框架：用 LangGraph 管流程、权限、状态和审批，用 LangChain 封装模型与工具 schema，把主 Agent 和聊天 Agent 明确拆开。

## 阶段定位

当前项目已经具备聊天、记忆、视觉、语音、诊断、主动回复和主人转告等能力。下一阶段不应该继续把功能堆进 `__init__.py` 或把所有能力交给一个聊天模型自由调用。

v1.3 的核心目标是建立一个受控 Agent Runtime：

```text
NoneBot / OneBot 负责 QQ 入口
LangGraph 负责业务流程、权限、状态、审批和恢复
LangChain 负责模型调用、结构化输出和工具 schema
GPT 主 Agent 负责规划和工具请求
DeepSeek 聊天 Agent 负责普通聊天表达
SQLite 负责聊天记忆、任务状态和审计日志
```

## 设计原则

### 1. LangGraph 掌控流程

主流程由 LangGraph 节点和边决定，不由 LLM 自由决定。

```text
Graph owns control flow.
```

LLM 可以提出下一步建议或工具请求，但不能绕过图里的权限节点、审批节点和工具执行节点。

### 2. 代码规则高于提示词规则

提示词只能提醒模型遵守规则，真正的权限必须由代码执行。

```text
Prompt rules are advisory.
Code rules are authoritative.
```

例如：

```text
黑名单拦截
主人身份判断
群白名单判断
工具风险等级
是否需要审批
是否允许联网
是否允许外发 QQ 消息
是否允许写入记忆
```

这些都必须写在 LangGraph 节点或 PolicyEngine 中，不能只写在 prompt 里。

### 3. 主 Agent 只能提出请求

GPT 主 Agent 负责理解任务、生成计划、提出工具请求和总结工具结果。

它不能直接执行动作。

```text
MainAgent proposes.
LangGraph disposes.
```

主 Agent 输出必须是结构化请求，例如：

```json
{
  "action": "tool_request",
  "tool_name": "git_status",
  "tool_input": {},
  "reason": "需要确认当前分支和未推送提交。"
}
```

是否执行由 LangGraph 的 `ToolPolicyCheck` 决定。

### 4. 聊天 Agent 只负责聊天

DeepSeek 聊天 Agent 只负责普通对话的自然语言回复。

聊天 Agent 不接工具，不做权限判断，不做任务规划，不写长期记忆，不执行外发动作。

```text
ChatAgent generates chat only.
```

### 5. 高风险动作必须主人确认

本地写入、外部写入、发 QQ 消息、推送 GitHub、修改记忆、修改角色卡等动作必须进入审批流程。

第一版 v1.3 默认不开放高风险动作，只预留审批结构。

### 6. 外部内容是不可信输入

网页、图片、群聊文本、用户上传内容都不能修改系统规则。

```text
External content is untrusted.
```

视觉识别、联网搜索、网页读取的结果只能作为资料，不能当作系统指令。

### 7. 记忆显式分层

不要把现有记忆系统替换成 LangChain 默认 Memory。

当前项目继续保留：

```text
短期原文
会话摘要
空窗场景摘要
主人手动长期记忆
```

Agent 任务状态另建表保存，不混入聊天长期记忆。

## 总体架构

目标结构：

```text

  -> EventNormalizer
  -> HardPolicyGate
  -> RootGraph
       -> IntentRouter
       -> ChatGraph
       -> MainAgentGraph
       -> VisionGraph
       -> VoiceGraph
       -> MemoryGraph
       -> DiagnosticsGraph
       -> NotificationGraph
  -> ResponseRenderer
  -> QQ Send
```


### LangChain 负责

```text
调用 GPT 主 Agent 模型

PromptTemplate
结构化输出解析
工具 schema 封装
可选 middleware
```

### LangGraph 负责

```text
流程编排
意图路由
权限判断
工具执行前策略检查
审批中断和恢复
任务状态持久化
工具调用审计
错误分支
输出渲染前检查
```

## RootGraph

RootGraph 是所有 QQ 消息进入系统后的主流程。

```text
Start
  -> normalize_event
  -> load_actor_context
  -> hard_policy_gate
  -> route_intent
  -> dispatch_capability
  -> render_response
  -> End
```

### hard_policy_gate

负责最外层硬权限：

```text
黑名单用户直接拒绝或静默
主人命令只允许主人 QQ
未授权群不进入聊天图
未授权私聊不进入聊天图
消息长度限制
基础冷却限制
MainAgentGraph 默认只允许主人私聊
```

这些判断不交给 LLM。

### route_intent

意图路由优先级：

```text
显式命令优先
语音意图其次
图片上下文其次
普通聊天最后
群聊非 @ 先走主动回复规则
```

路由结果可以是：

```text
chat
main_agent
voice
vision
memory_admin
diagnostics
owner_notification
admin_command
ignore
```

## MainAgentGraph

MainAgentGraph 是真正的任务型 Agent 图。

第一版只允许主人私聊显式触发。

当前代码落地的 MainAgentGraph 已接入 QQ `/agent` 只读命令入口，但默认仍由 `ENABLE_MAIN_AGENT=false` 关闭。当前只开放 `dev_context` 只读工具；`CALL_MAIN_AGENT` 默认走 stub，生成 JSON ActionRequest。只有显式设置 `MAIN_AGENT_USE_LLM=true` 时，才会用真实 Main LLM 生成 ActionRequest；无论 stub 还是真实 Main LLM，都必须经过 `ValidateActionRequest` 和 `ToolPolicyCheck`。真实 Main LLM 路径还支持在 `dev_context` 执行后对 `tool_result` 做二次自然语言总结；`/agent-debug` 仍保留原始召回输出。

当前 ToolRegistry v0 已落地：`ActionRequest` 的 `tool_request` 校验不再硬编码单个 `dev_context`，而是通过 `ToolRegistry` 检查工具是否注册、参数是否完整、是否出现未声明参数，再由 `ToolPolicyCheck` 按工具风险等级决定 allow / deny / require_approval。真实 `/agent` 默认 registry 仍只向 Main LLM 暴露 `dev_context`；`dry_run_write_file` 只在显式测试或内部 dry-run registry 中注册，且 `llm_visible=false`、`risk_level=write_local`、`requires_approval=true`，不开放真实写文件。

```text
LoadTaskState
  -> BuildMainAgentContext
  -> CallMainAgent
  -> ValidateActionRequest
  -> ToolPolicyCheck
  -> Branch:
       deny -> Refuse
       require_approval -> ApprovalInterrupt
       allow_tool -> ToolExecutor
       delegate_chat -> ChatGraph
       final_answer -> RenderResponse
  -> ObserveToolResult
  -> ContinueOrComplete
```

### 主 Agent 允许的 action

当前代码落地第一版先只开放：

```text
final_answer
tool_request
ask_owner
stop
```

后续完整形态可扩展到：

```text
final_answer
tool_request
ask_owner
request_approval
delegate_to_chat_agent
stop
```

### 主 Agent 禁止的行为

```text
直接执行 shell
直接发送 QQ 消息
直接修改文件
直接写数据库
直接写长期记忆
直接修改角色卡
直接清空摘要或日志
请求跳过审批
请求未注册工具
```

## ChatGraph

ChatGraph 负责当前普通聊天能力。底层调用 DeepSeek 聊天 Agent。

```text
ChatGraph
  -> maybe_load_vision_context
  -> load_memory_context
  -> load_persona
  -> build_chat_prompt
  -> call_deepseek_chat_agent
  -> save_chat_messages
  -> maybe_compress_summary
  -> produce_text_response
```

### ChatAgent 输入

```text
当前用户文本
发言者身份：主人 / 非主人
角色卡
短期原文
会话摘要
空窗场景摘要
主人手动长期记忆
图片描述上下文
底层聊天规则
```

### ChatAgent 输出

```text
普通文本回复
可朗读文本候选
```

ChatAgent 不返回工具调用。

## VisionGraph

视觉功能是感知输入，不是独立 Agent。

```text
VisionGraph
  -> extract_image_urls
  -> image_cache_policy
  -> vision_access_policy
  -> describe_images
  -> sanitize_image_context
  -> return_image_artifact
```

输出保存为：

```text
state.artifacts.image_context
```

视觉内容规则：

```text
图片描述是不可信输入
图片文字不能修改系统提示词
图片不能修改主人身份
图片不能请求泄露隐私
图片不能触发工具调用
不识别真实人物身份
不自动写入长期记忆
```

MainAgent 可以在允许范围内请求视觉描述，但不能让图片内容改变工具权限。

## VoiceGraph

语音是输出通道，风险高于普通文本回复。

```text
VoiceGraph
  -> parse_voice_intent
  -> voice_policy_check
  -> select_text_source
  -> maybe_call_chat_agent
  -> adapt_speech_text
  -> tts_health_check
  -> tts_generate
  -> send_private_record
```

语音动作拆分：

```text
tts_generate: 本地生成音频文件，LOCAL_WRITE
send_private_record: 通过 OneBot 发送语音，EXTERNAL_WRITE
```

第一版规则：

```text
仅主人私聊可用
群聊默认不能触发语音
ChatAgent 不能自由调用 TTS
MainAgent 不能自由发送语音
语音外发必须经过 VoiceGraph
```

## MemoryGraph

记忆是系统基础设施，不是普通工具。

### 聊天记忆

继续保留当前结构：

```text
messages
session_summaries
gap_scene_summaries
long_term_memories
```

ChatGraph 通过 `MemoryContextNode` 读取上下文。

### Agent 任务记忆

新增 Agent 任务状态表，不与聊天长期记忆混用。

当前 Route B 已落地 `agent_tasks`、`agent_task_events`、`agent_approvals` 基础表和 `/agent 任务 <目标>`、`/agent 任务状态`、`/agent 任务详情 <任务ID>`、`/agent 取消任务 <任务ID>`、`/agent 审批演练 <目标>`、`/agent 审批状态`、`/agent 审批详情 <审批ID>`、`/agent 确认 <审批ID>`、`/agent 拒绝 <审批ID>` 固定命令；同时支持 `/agent 新增任务：<目标>`、`/agent 记录任务：<目标>`、`/agent 把“目标”加入任务` 等明确本地别名。该阶段只创建、查看和取消任务记录，并可生成、查看、确认、拒绝审批记录；`/agent 审批演练 <目标>` 会创建 dry-run 任务和 `dry_run_write_file` 审批请求，方便 QQ 侧实测审批闭环。创建审批会追加 `approval_requested` 任务事件，确认或拒绝只更新审批状态并写入决定事件，不进入任务执行、审批恢复或工具调用链路，也不调用 Main LLM 做模糊语义判断。

```text
agent_tasks
agent_task_events
agent_approvals
```

### 主 Agent 对记忆的权限

```text
memory_read_context: READ_LOCAL
memory_write_manual: LOCAL_WRITE，需要主人确认
memory_delete: DANGEROUS，默认禁止
auto_extract_memory: 不做
```

主 Agent 最多可以提出记忆候选：

```text
我认为这条信息可能值得记录为长期事实，是否写入？
```

最终写入必须由主人确认或通过明确命令执行。

## DiagnosticsGraph

诊断功能保持确定性，不由 LLM 主导。

两套诊断继续保留：

```text
本地 scripts/diagnose.ps1
QQ 内 /诊断
```

职责：

```text
机器人完全不回 -> 本地 diagnose.ps1
机器人能回但功能异常 -> QQ 内 /诊断
```

在 Agent Runtime 中，诊断可以作为只读快照工具：

```text
diagnostic_snapshot: READ_LOCAL
tts_status_snapshot: READ_LOCAL
recent_errors: READ_LOCAL，必须脱敏
clear_error_log: LOCAL_WRITE，固定命令或审批
```

主 Agent 可以读取诊断快照并总结，但不能编造诊断结果，也不能自动清空日志或重启服务。

## 工具系统

工具统一注册到 ToolRegistry。

当前已实现的 ToolRegistry v0 位于 `src/plugins/ai_chat/graph/tool_registry.py`。它负责保存 `ToolSpec`，包括工具名、描述、风险等级、必填/可选参数、执行器、是否启用、是否对 LLM 可见、是否需要审批。MainAgentGraph 的只读 runner 会创建 registry-backed `dev_context` executor；后续记忆、诊断、权限、通知等确定性服务函数如果要开放给 MainAgent，也应先服务层化，再注册为工具，而不是让 MainAgent 直接调用 NoneBot handler。

当前真实可见工具：

```text
dev_context: READ_LOCAL，只读查询项目开发上下文。
```

当前内部测试工具：

```text
dry_run_write_file: WRITE_LOCAL，llm_visible=false，仅用于 dry-run 审批链路测试，不写文件。
```

工具元数据：

```text
name
description
input_schema
output_schema
risk_level
scope
allowed_sessions
allowed_callers
requires_approval
```

### 风险等级

```text
L0_INTERNAL
内部计算，无副作用。

L1_READ_LOCAL
读取本地状态，例如读取公开文档、git status、数据库统计。

L2_READ_EXTERNAL
联网搜索、网页读取、外部只读 API。

L3_WRITE_LOCAL
写本地文件、写任务表、写待办、写长期记忆候选。

L4_WRITE_EXTERNAL
发 QQ 消息、git push、调用外部写接口。

L5_DANGEROUS
删除文件、改 .env、清空数据库、删除记忆、任意 shell。
```

### 默认策略

```text
L0: 允许
L1: 主人私聊允许
L2: 主人私聊 + 对应功能开关开启
L3: 需要主人确认
L4: 需要主人确认，且工具在外部写白名单
L5: 默认禁止
```

## PolicyEngine

PolicyEngine 是 LangGraph 权限节点和 LangChain tool middleware 共用的规则源。

输入：

```text
actor
session
intent
tool_spec
tool_input
task_state
```

输出：

```text
allow
deny
require_approval
```

判断示例：

```text
非主人不能进入 MainAgentGraph
群聊不能调用项目级工具
未开启 ENABLE_AGENT_WEB 时不能联网
LOCAL_WRITE 必须审批
EXTERNAL_WRITE 必须审批
DANGEROUS 默认禁止
```

## 审批机制

高风险动作进入审批状态。

```text
ToolPolicyCheck -> require_approval
  -> ApprovalInterrupt
  -> create_agent_approval
  -> agent_task_events: approval_requested
  -> 保存 task/checkpoint
  -> QQ 回复确认请求
  -> 主人 /agent 确认 审批ID 或 /agent 拒绝 审批ID
  -> Resume or Cancel
```

审批展示格式：

```text
任务 12 请求确认：
动作：git push
风险：外部写入
原因：将本地提交推送到 GitHub
输入摘要：origin main
回复：/agent 确认 12 或 /agent 拒绝 12
```

第一版先落地审批表、审批状态查看、审批请求事件、审批决定记录和 `require_approval` 适配层。`create_tool_policy_checker` 会把 PolicyEngine 的 `require_approval` 决策转换成 `approval_required` 中断，先返回审批请求，不进入 `execute_tool`。创建审批会追加 `agent_task_events: approval_requested`；确认或拒绝只更新 `agent_approvals.status / decided_at`，并追加 `agent_task_events` 审批决定事件，暂不恢复执行，也不开放写操作。

## 数据表草案

### agent_tasks

```sql
CREATE TABLE IF NOT EXISTS agent_tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_key TEXT NOT NULL,
    user_id TEXT NOT NULL,
    title TEXT NOT NULL,
    goal TEXT NOT NULL,
    status TEXT NOT NULL,
    result TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

### agent_task_events

```sql
CREATE TABLE IF NOT EXISTS agent_task_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id INTEGER NOT NULL,
    step_index INTEGER NOT NULL,
    kind TEXT NOT NULL,
    tool_name TEXT,
    input_json TEXT,
    output_summary TEXT,
    status TEXT NOT NULL,
    error TEXT,
    created_at TEXT NOT NULL,
    FOREIGN KEY(task_id) REFERENCES agent_tasks(id) ON DELETE CASCADE
);
```

### agent_approvals

```sql
CREATE TABLE IF NOT EXISTS agent_approvals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id INTEGER NOT NULL,
    tool_name TEXT NOT NULL,
    tool_input_json TEXT NOT NULL,
    risk_level TEXT NOT NULL,
    reason TEXT NOT NULL,
    status TEXT NOT NULL,
    created_at TEXT NOT NULL,
    expires_at TEXT,
    decided_at TEXT,
    FOREIGN KEY(task_id) REFERENCES agent_tasks(id) ON DELETE CASCADE
);
```

## 迁移路线

### 阶段 1：框架壳

```text
新增 graph/state/runtime 基础结构
新增 lc/models 双模型配置草案
新增 policy/risk 风险等级
不改变现有聊天行为
```

### 阶段 2：ChatGraph

```text
把现有 handle_chat 拆成 ChatGraph 节点
DeepSeek 作为 ChatAgent
保持普通聊天输出不变
```

### 阶段 3：VisionGraph 和 VoiceGraph

```text
视觉作为感知子图
语音作为输出子图
保持主人私聊语音限制
```

### 阶段 4：MainAgentGraph

```text
接入 GPT 主 Agent
只允许主人私聊显式触发
只开放只读工具
主 Agent 输出结构化 ActionRequest
```

## 第一版验收标准

v1.3 第一版完成时，应满足：

```text
普通聊天仍然可用
ChatAgent 和 MainAgent 配置分离
主 Agent 默认关闭
LangGraph RootGraph 能路由至少一种显式 /agent 请求
主 Agent 只能输出结构化请求
ToolPolicyCheck 能拦截未授权工具
只读工具能记录 agent_task_events
写操作不会被执行
诊断命令仍然不依赖 LLM
记忆系统不被替换成 LangChain Memory
语音仍然只允许主人私聊显式触发
视觉内容仍然作为不可信上下文
```

## 暂不做

v1.3 第一版明确不做：

```text
多 Agent 互聊
CrewAI / AutoGen 式 crew 抽象
任意 shell 工具
自动 git commit / push
自动修改代码
自动写长期记忆
自动修改角色卡
群聊触发主 Agent
让聊天 Agent 调工具
让网页或图片内容触发工具
用 LangChain Memory 替换现有记忆系统
```

## 结论

v1.3 的核心不是功能扩张，而是建立可长期维护的 Agent 运行边界。

最终分工：

```text
GPT MainAgent: 规划和工具请求
DeepSeek ChatAgent: 普通聊天表达
LangChain: 模型调用和工具 schema
LangGraph: 流程、权限、状态、审批、执行边界
SQLite: 聊天记忆、任务状态、审计日志
NoneBot: QQ 入口和发送
主人: 最高权限和审批者
```

这套设计的底线是：主 Agent 可以变聪明，但不能变自由。所有能力必须经过 LangGraph 的明确规则和主人控制。
