# P2.48 Owner Console 可靠性可用性与演进设计

状态：现状审计与设计已完成；P2.48a 最小可读性优化已实现、加载，并由主人在真实页面确认可读性很高，live 验收通过。

设计日期：2026-07-21。

基线提交：`8f34819 Complete P2.47 reliability and pure-image workflows`。

## 1. 设计结论

P2.48 的第一优先级不是继续增加更多表格、筛选项或 external-read 时间线，而是给现有结构化可靠性数据增加一层真正面向主人排障的解释层：

```text
先用中文回答当前是否有需要关注的问题。
再解释哪些故障已恢复、哪些仍未恢复、哪些证据不足。
每条结论必须能回到固定结构化事件证据。
筛选和技术表格降为下钻工具，不再承担主要阅读任务。
自然总结由固定规则生成，不调用 LLM，不猜测根因。
```

Owner Console 现有可靠性页在合同正确性、只读边界和筛选准确性上已经合格，但信息架构仍偏“开发者验证页面”：六个同权指标、三个筛选器和七列宽表格占据主要空间，主人需要自行翻译 component、operation、code 和恢复状态之间的关系，才能判断应关注什么。P2.48 应把它改造成“结论优先、证据可追溯、技术细节可下钻”的只读工作台。

原先候选方向中的 external-read 时间线暂不进入第一刀。当前 external-read 读模型只有最近一次正式任务安全快照，没有已经评审的多任务历史合同，也没有明确到“主人希望通过时间线回答什么问题”的使用场景。此时直接增加时间线会扩大数据面，却未必提高排障效率。它应在可靠性解释层完成后，满足本文第 13 节前置条件时再决定是否实现。

## 2. 本轮审计范围与限制

本轮只读检查了：

```text
Owner Console 可靠性 React 页面。
可靠性页面 CSS 与窄屏规则。
TypeScript HTTP DTO。
Python read model 与趋势 builder。
P2.47 结构化趋势和 QQ 文本 formatter。
可靠性页 Vitest 与后端 read-model 测试。
external-read 页面当前数据结构。
既有 UI 布局、只读 read model 和受控自动刷新设计。
当前 8090 监听状态。
```

审计时 Owner Console 未在 8090 监听，因此没有擅自启动或重启服务，也没有声称读取了本轮实时生产快照。当前工具会话虽然列出 Browser 技能，但没有暴露技能要求的内置浏览器控制接口；按技能边界没有使用外部 Playwright 或其他浏览器自动化绕过。视觉可读性结论来自页面源码、样式、现有自动化、P2.47 已完成的主人人工页面验收，以及主人本轮明确反馈。P2.48 实现完成后仍必须重新进行真实浏览器桌面端与移动端人工验收。

## 3. 当前页面保留价值

现有实现中以下能力具有明确实用价值，应保留：

```text
最近 24 小时与最近 7 天两个固定窗口。
失败/降级 occurrence_count 与故障组数量分离。
unresolved、recovered、recurring、insufficient_evidence 四种严格状态。
最近成功只在具备可靠成功语义时展示。
组件、类别和恢复状态的纯前端本地筛选。
筛选不产生额外后端请求或写入。
当前接入范围和只读边界可核对。
SQLite mode=ro，不调用 ensure_database。
不返回正文、用户、QQ、路径、URL、Key、原始异常或 runtime/database ID。
```

这些是 P2.47 已验证的底层事实。P2.48 不重写事件合同，不修改恢复推导，也不通过展示层伪造新的故障结论。

## 4. 实用性审计

### 4.1 当前能回答的问题

当前页面可以回答：

```text
窗口内有多少失败/降级记录。
聚合成多少个故障组。
每组属于哪个 component/operation/category/code。
每组首次和最后失败时间。
是否存在最后失败之后的真实成功证据。
筛选后还剩哪些固定故障组。
```

### 4.2 当前难以回答的问题

当前页面不能直接回答：

```text
现在最需要主人关注的是哪一组。
某个英文 component/operation 对应什么实际功能。
稳定 code 的中文含义是什么。
“证据不足”和“未恢复”在行动上有什么区别。
已恢复是否代表历史故障可以忽略。
下一步应先检查服务、模型、配置还是数据。
筛选后这些结果共同说明了什么。
24 小时与 7 天之间能否证明故障正在增加或下降。
页面上的快照是否只是长时间未手动刷新。
```

### 4.3 当前低收益区域

六个指标卡片同权排列，会把“失败次数”“故障组”“已恢复”“证据不足”等不同语义压成一组数字，主人仍需自己解释。底部七项接入范围和只读边界对合同验收有价值，但在日常排障中不应占据与故障结论相同的视觉优先级。

当前生产数据通常只有少量故障组。三个筛选器虽然正确，但当只有四到五组数据时，筛选本身不是页面的核心价值。筛选只有在中文化并能同步更新解释摘要后，才真正帮助排障。

external-read 当前只有最近任务快照。在没有多条安全历史记录、没有已定义排序/分页/保留期限、没有明确排障问题前，时间线属于“看起来完整但实际用途不确定”的功能，第一刀不做。

## 5. 可读性审计

当前页面的主要可读性问题：

```text
component、operation 和 code 以英文内部标识作为主文案。
只有 category_label 和 recovery_state_label 完成中文化。
七列证据表最小宽度为 1080px，窄屏主要依赖横向滚动。
同一行同时出现组件、操作、类别、代码、三个时间和状态，扫描成本高。
恢复状态虽然是中文，但没有就地解释其证据语义。
表格默认按 occurrence_count 和最后失败时间排序，但页面没有说明优先级。
筛选结果只显示“X/Y 组”，没有解释筛选后最值得关注的结论。
“没有结构化故障不等于持续在线”的限制说明离主要结论较远。
```

既有 UI 设计早已规定“后端字段保留英文，页面主要展示中文；技术字段只在详情或 tooltip 中保留原值”。可靠性页目前没有完全贯彻该原则。P2.48 必须修正这一点，但不能修改底层 component、operation、category、code 的稳定英文合同。

## 6. P2.48 信息架构

推荐页面顺序：

```text
+------------------------------------------------------+
| 页面标题 | 快照生成时间 | 手动刷新/受控自动刷新状态 |
+------------------------------------------------------+
| 当前结论：需要关注 / 已恢复为主 / 证据有限 / 无记录 |
| 2-4 句确定性中文自然总结                            |
+------------------------------------------------------+
| 重点问题（最多 3 组）                               |
| 中文功能 + 中文问题 + 可证状态 + 最近证据 + 下一检查 |
+------------------------------------------------------+
| 24 小时 / 7 天 | 需要关注 | 已恢复 | 证据不足       |
+------------------------------------------------------+
| 中文筛选：功能、问题类别、证据状态 | 清除筛选        |
| 筛选后的局部总结                                    |
+------------------------------------------------------+
| 完整证据明细：默认精简；技术标识可展开              |
+------------------------------------------------------+
| 接入范围与只读安全边界：次级、可折叠或紧凑展示      |
+------------------------------------------------------+
```

页面必须让主人在不阅读表格的情况下先回答：

```text
是否有需要关注的问题。
最重要的两到三组是什么。
为什么被判断为未恢复、反复、已恢复或证据不足。
下一步应该检查哪一层，但系统没有自动执行什么。
```

## 7. 中文标签合同

接口仍返回英文稳定值；后端 presentation builder 同时返回固定中文标签。前端不得自行散落多套翻译表。

### 7.1 component 中文标签

| 稳定值 | 主要中文标签 |
| --- | --- |
| `bot_runtime` | Bot 运行状态 |
| `main_llm` | MainAgent 规划模型 |
| `sticker_classifier` | 表情意图分类 |
| `document_delivery` | 文档交付 |
| `project_doc_rag` | 项目文档索引 |
| `vision` | 图片理解 |
| `tts` | 语音合成 |

### 7.2 operation 中文标签

| 稳定值 | 主要中文标签 |
| --- | --- |
| `lifecycle` | 进程启动与停止 |
| `plan_action` | 生成行动计划 |
| `classify_intent` | 判断表情意图 |
| `send_document` | 发送新文档 |
| `rebuild_index` | 重建项目文档索引 |
| `infer` | 识别图片内容 |
| `synthesize` | 生成语音 |

页面主标题使用中文，例如“图片理解 · 识别图片内容”。原始 `vision / infer` 放在次级技术详情中，仍可复制和审计。

### 7.3 code 中文解释

| 稳定 code | 中文解释 |
| --- | --- |
| `suspected_abnormal_exit` | 上次运行没有发现正常停止记录 |
| `request_timeout` | 请求在限定时间内没有完成 |
| `connection_failed` | 服务连接或传输失败 |
| `model_rate_limited` | 模型服务拒绝或限制了当前请求 |
| `model_not_found` | 配置的模型当前不可用 |
| `invalid_model_response` | 模型响应未通过格式或质量校验 |
| `authorization_failed` | 鉴权或权限检查未通过 |
| `invalid_configuration` | 功能配置不完整或不合法 |
| `data_validation_failed` | 输入或产物未通过数据校验 |
| `presentation_slide_limit_exceeded` | 演示文稿页数超过安全上限 |
| `artifact_integrity_failed` | 文档产物完整性复核失败 |
| `document_delivery_failed` | 文档交付没有完成 |
| `approval_context_invalid` | 审批上下文与当前任务不一致 |
| `required_arguments_unavailable` | 执行所需参数不完整 |
| `unexpected_runtime_state` | 运行状态不在已知安全范围内 |

中文解释只能描述已验证到的失败边界，不能把 `request_timeout` 扩写为“网络中断”，也不能把 `invalid_model_response` 擅自细化为空响应、JSON 错误、重复内容或超长响应。

### 7.4 恢复状态解释

| 状态 | 页面解释 | 排障含义 |
| --- | --- | --- |
| `unresolved` | 最后一次失败之后没有发现真实成功证据 | 需要关注 |
| `recurring` | 期间出现过成功，但最后状态又回到失败 | 优先关注反复问题 |
| `recovered` | 最后一次失败之后出现了真实成功证据 | 保留历史证据，当前不列为未恢复 |
| `insufficient_evidence` | 该 operation 不具备可据此证明恢复的成功语义 | 只能说明证据不完整，不能判定未恢复或已恢复 |

“已恢复”不能写成“根因已永久解决”；“证据不足”不能用危险红色，也不能等同于未恢复。

## 8. 确定性自然总结设计

### 8.1 不使用 LLM

自然总结必须由固定结构化数据和模板生成：

```text
不调用 Main LLM、Chat LLM、Tavily、MemoryRAG 或 ProjectDocRAG。
不读取聊天正文、原始异常或用户标识。
不产生新的可靠性事件。
不执行诊断、探针、重试、重启、告警或修复。
同一输入必须得到同一结论。
每个句子必须能追溯到一个或多个固定趋势项。
```

现有 `format_reliability_trend_report` 已能生成 QQ 中文列表，但仍以英文 component/operation/code 为主要内容，也没有形成“需要关注什么”的解释层。P2.48 应抽出共享的 presentation builder，让 QQ formatter 和 Owner Console read model 复用同一份中文标签和证据判断，而不是让前端解析 QQ 文本。

### 8.2 建议 read model

```text
OwnerConsoleReliabilityNarrative
  level: stable | attention | recurring | limited_evidence
  headline: str
  summary_lines: list[str]
  findings: list[OwnerConsoleReliabilityFinding]
  limitations: list[str]

OwnerConsoleReliabilityFinding
  evidence_key: component + operation + category + code
  component_label: str
  operation_label: str
  code_label: str
  recovery_state: str
  recovery_state_label: str
  conclusion: str
  evidence_line: str
  next_check: str
```

`next_check` 只能来自固定类别建议：

```text
配置问题：检查功能开关、必填配置、服务地址和模型名。
模型问题：检查模型是否存在、额度/限流和响应格式，避免立即连续重试。
权限问题：检查鉴权、主人权限或审批状态，不绕过权限。
网络问题：检查本地服务和网络可达性，再由主人决定是否重试。
数据问题：检查数据库、索引、文件完整性和输入格式，不覆盖原数据。
```

### 8.3 优先级

重点问题最多展示三组，排序规则固定为：

```text
1. recurring。
2. unresolved。
3. insufficient_evidence。
4. recovered。
同状态内按 occurrence_count 降序。
再按 last_seen_at 降序。
最后按稳定 component/operation/category/code 排序。
```

`recurring` 优先于 `unresolved`，因为它表示问题在出现成功后再次失败；`insufficient_evidence` 不得压过真正未恢复故障；`recovered` 主要用于说明历史问题已有后续成功证据。

### 8.4 固定总结场景

无故障组：

```text
最近 24 小时没有记录到已接入组件的结构化失败或降级事件。
这不等于系统持续在线，也不覆盖尚未接入的组件。
```

只有 recovered：

```text
最近 24 小时记录到 N 组历史故障，所有故障在最后失败之后都有真实成功证据。
当前没有结构化未恢复组，但历史失败仍保留在证据中。
```

存在 unresolved：

```text
最近 24 小时有 N 组需要关注的问题，其中 M 组在最后失败之后没有成功证据。
优先检查：<中文功能与问题>。
```

存在 recurring：

```text
最近 24 小时发现反复发生的问题：<中文功能与问题>。
它曾出现成功证据，但最后一次记录再次失败，应优先排查波动原因。
```

只有 insufficient_evidence：

```text
当前没有可判定为未恢复的故障组，但有 N 组证据不足。
这些记录不能证明故障仍在，也不能证明已经恢复。
```

混合状态时先说需要关注项，再说已恢复项，最后说明证据不足和覆盖边界。

### 8.5 不伪造“上升/下降趋势”

当前 API 只有“以现在为终点的最近 24 小时”和“最近 7 天累计”。不能把 24 小时次数直接与 7 天总数比较后写成“上升”或“下降”，因为窗口长度不同。

P2.48 第一刀中的“自然趋势总结”只解释当前窗口的故障结构、恢复状态和证据，不生成环比方向。如果未来需要“较上一周期增加/减少”，必须新增等长比较合同，例如：

```text
current_24h: [now-24h, now]
previous_24h: [now-48h, now-24h)
```

或者提供固定日桶时间序列。只有在等长窗口、边界不重叠、接入范围一致时，页面才可以使用“增加、下降、持平”等词。

## 9. 中文筛选设计

筛选值继续使用稳定英文 enum，选项展示使用固定中文标签：

```text
功能：全部功能 / Bot 运行状态 / MainAgent 规划模型 / 表情意图分类 / ...
问题类别：全部类别 / 配置问题 / 模型问题 / 权限问题 / 网络问题 / 数据问题
证据状态：全部状态 / 需要关注 / 反复发生 / 已恢复 / 证据不足
```

建议把“恢复状态”改名为“证据状态”，因为 `insufficient_evidence` 并不是一种恢复结果。

筛选仍为纯前端本地行为，不产生新 API 请求。筛选后：

```text
自然总结、重点问题卡片和证据明细必须基于同一 filteredRows。
页面明确显示“当前筛选 X 组，共 Y 组”。
清除筛选在全部为默认值时禁用。
技术原值不出现在 select 主标签中。
```

可以增加一个有实际意义的快捷筛选“只看需要关注”，其定义严格为 `unresolved + recurring`。第一刀不增加自由文本搜索、复杂查询表达式、保存筛选、URL 持久化或跨页面筛选状态。

## 10. 指标与证据明细重排

六个同权指标建议改为四个主要指标：

```text
失败/降级记录次数。
需要关注组数（unresolved + recurring）。
已有恢复证据组数。
证据不足组数。
```

故障组总数作为辅助文案展示，不再占一个同权卡片。“失败/降级次数”应改成“失败/降级记录次数”，并说明它是五分钟桶聚合后的 occurrence_count，不等于独立事故数量。

完整证据明细保留，但主视图只显示：

```text
中文功能与操作。
中文问题解释。
次数。
最后失败时间。
中文证据状态。
```

首次失败、最近成功、原始 component/operation/category/code 放入展开详情。桌面端可以保留紧凑表格；移动端优先使用纵向证据卡片，避免把 1080px 宽表格作为唯一阅读方式。

## 11. 快照新鲜度设计

必须区分两个概念：

```text
页面快照多久没有刷新。
某个按需组件多久没有产生事件。
```

`generated_at` 是每次 GET 生成快照的时间。页面长时间停留后，可以提示“此页面快照可能已过期，请手动刷新”，这是页面新鲜度。

vision、tts、document_delivery、sticker_classifier 等组件按需运行。长时间没有事件可能只是没有真实用户请求，不能被标记为“组件 stale”或“不健康”。Bot lifecycle 也可能只在启动/停止时产生事件。P2.48 不基于最后事件时间自动判定服务过期。

第一刀建议：

```text
始终明确显示快照生成时间。
说明可靠性页默认是手动快照，不等于实时监控。
如果复用全局受控自动刷新，必须使用现有 useControlledAutoRefresh。
页面不得自行创建 setTimeout/setInterval 或 visibility 监听。
自动刷新仍由主人显式打开，默认关闭。
刷新失败时保留最近一次成功数据，并显示“快照可能过期”。
```

不建议仅为了倒计时新增页面私有 timer。如果后续需要动态 stale 横幅，应扩展现有受控刷新基础设施或统一状态条，继续通过 GET-only guard。

## 12. 响应式与可读性验收

实现阶段必须验证：

```text
桌面宽屏：1440px。
普通桌面：1024px。
平板/窄窗口：768px。
手机：430px 和 360px。
```

验收重点：

```text
自然总结不出现超宽长行。
重点问题卡片按单列降级。
筛选器在窄屏纵向排列，按钮占满可用宽度。
```

## 13. external-read 时间线的进入条件

```text

不返回 query、结果正文、URL、来源标题、用户/QQ、Key 或原始异常。
```

在此之前，external-read 页面保留“最近正式任务”即可。优先把当前已有可靠性数据解释清楚。

## 16. 实现验收标准

P2.48a-b 只有同时满足以下条件才算完成：

```text

11. API 和 read model 继续不含正文、身份、秘密、路径和原始异常。
12. 所有 HTTP 请求继续经过 GET allowlist，web_write_enabled=false。

14. 自动化、真实页面和主人可读性验收同时通过。
```
