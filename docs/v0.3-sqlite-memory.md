# v0.3 SQLite 记忆方案

## 目标

v0.3 先解决两个轻量但关键的问题：

1. 聊天上下文从内存迁移到 SQLite，机器人重启后仍然能保留近期上下文。
2. 陌生人私聊试用次数从内存迁移到 SQLite，重启后不会被重置。

同时，v0.3 会把数据库结构设计成分层记忆架构，方便后续继续升级。

## 分层记忆架构

最终记忆系统分三层。

第一层：短期对话缓存

```text
用途：保持当前私聊或当前群聊的连续上下文
数据：最近若干轮 user / assistant 消息
读写：每次对话都会读取和写入
当前状态：v0.3 已实现
```

第二层：长期记忆归档

```text
用途：保存稳定事实、用户偏好、用户性格特点、群聊设定和重要事件
数据：例如“用户喜欢简洁回复”“用户正在做 QQ AI 机器人项目”
读写：后续由 AI 总结器或主人命令写入
当前状态：v0.3 先预留数据库表，暂不自动写入
```

第三层：语义索引

```text
用途：当用户提到以前的事时，按语义相似度召回相关长期记忆
数据：长期记忆对应的 embedding 向量
读写：后续接入 embedding 模型后生成和检索
当前状态：v0.3 先预留数据库表，暂不接向量检索
```

这样拆分的好处是：

- 短期对话负责“接上刚刚说的话”。
- 长期记忆负责“记住这个人和这个群的重要事实”。
- 语义索引负责“从很多旧记忆里找回相关内容”。
- 后续升级时可以逐层启用，不会影响当前稳定聊天。

## 选型

数据库文件：

```text
data/chatbot.db
```

选择 SQLite 的原因：

- 不需要单独安装数据库服务，适合 Windows 本地运行。
- Python 标准库自带 `sqlite3`，不需要新增依赖。
- 对当前 QQ 机器人这种单机轻量场景足够稳定。
- 文件在 `data/` 目录内，已经被 `.gitignore` 忽略，不会误提交聊天记录。

暂不选择 MongoDB、MySQL、Pinecone、ChromaDB 或 DuckDB：

- MongoDB/MySQL 更适合多服务、多用户后台系统，目前部署成本偏高。
- Pinecone/ChromaDB 更适合语义检索和长期知识库，当前阶段还不需要。
- DuckDB 更适合分析型查询，不适合承担聊天机器人的在线状态存储。

## 数据表

### messages

保存用户和 AI 的聊天消息。

```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_key TEXT NOT NULL,
    message_type TEXT NOT NULL,
    user_id TEXT NOT NULL,
    group_id TEXT,
    role TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at TEXT NOT NULL
);
```

字段说明：

- `session_key`: 会话标识，私聊为 `private:QQ号`，群聊为 `group:群号`。
- `message_type`: `private` 或 `group`。
- `user_id`: 发送消息的 QQ 号。
- `group_id`: 群聊时保存群号，私聊为空。
- `role`: `user` 或 `assistant`。
- `content`: 消息正文。
- `created_at`: 写入时间。

### private_trials

保存陌生人私聊试用次数。

```sql
CREATE TABLE private_trials (
    user_id TEXT PRIMARY KEY,
    used_count INTEGER NOT NULL DEFAULT 0,
    updated_at TEXT NOT NULL
);
```

### long_term_memories

保存长期记忆归档。

```sql
CREATE TABLE long_term_memories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    subject_type TEXT NOT NULL,
    subject_id TEXT NOT NULL,
    memory_type TEXT NOT NULL,
    content TEXT NOT NULL,
    source_session_key TEXT,
    confidence REAL NOT NULL DEFAULT 1.0,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

字段说明：

- `subject_type`: 记忆主体类型，例如 `user`、`group`、`bot`。
- `subject_id`: 主体 ID，例如 QQ 号或群号。
- `memory_type`: 记忆类型，例如 `preference`、`personality`、`fact`、`event`。
- `content`: 归档后的自然语言记忆。
- `source_session_key`: 这条记忆来自哪个会话。
- `confidence`: 可信度，后续用于过滤不确定记忆。

### memory_embeddings

保存长期记忆对应的语义向量。

```sql
CREATE TABLE memory_embeddings (
    memory_id INTEGER PRIMARY KEY,
    provider TEXT NOT NULL,
    model TEXT NOT NULL,
    embedding_json TEXT NOT NULL,
    created_at TEXT NOT NULL,
    FOREIGN KEY(memory_id) REFERENCES long_term_memories(id)
        ON DELETE CASCADE
);
```

当前先用 `embedding_json` 保存向量，适合轻量阶段。后续如果记忆数量变多，再考虑换成专门的向量数据库或 SQLite 向量扩展。

### schema_meta

保存数据库版本。

```sql
CREATE TABLE schema_meta (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

## 上下文策略

调用 AI 时只读取当前会话最近 `MAX_CONTEXT_MESSAGES` 条消息：

```text
SELECT role, content
FROM messages
WHERE session_key = ?
ORDER BY id DESC
LIMIT ?
```

读取后再按旧到新顺序传给模型。

这样可以让数据库保留较多历史，同时控制每次请求传给模型的上下文长度。

后续引入第二层和第三层后，完整上下文组装顺序建议为：

```text
系统提示词
  + 当前用户/当前群相关长期记忆
  + 语义索引召回的相关旧记忆
  + 当前会话短期上下文
  + 用户本次消息
```

这样可以让机器人既能接住当前对话，又不会把所有旧聊天记录都塞给模型。

## QQ 命令

普通命令：

```text
/重置
```

清空当前会话上下文。

主人命令：

```text
/记忆状态
/清空全部上下文
```

`/记忆状态` 查看数据库中的消息数量、会话数量和试用用户数量。
它也会显示长期记忆数量和语义索引数量。

`/清空全部上下文` 清空全部聊天上下文，但不清空白名单、黑名单和私聊试用次数。
它也不会清空长期记忆和语义索引。

## 隐私

SQLite 数据库会保存真实聊天内容，因此必须继续忽略：

```text
data/chatbot.db
```

当前 `.gitignore` 已经忽略 `data/*`，因此数据库不会被提交。

## 测试

基础测试：

1. 私聊连续对话，确认上下文正常。
2. 群聊 @ 机器人连续对话，确认上下文正常。
3. 重启 NoneBot2 后端，确认近期上下文仍然存在。
4. 使用 `/重置`，确认当前会话上下文被清空。
5. 主人使用 `/记忆状态`，确认能看到数据库统计。
6. 主人使用 `/清空全部上下文`，确认全部上下文被清空。
7. 陌生人私聊试用次数达到上限后，重启机器人仍然保持上限。
8. `/记忆状态` 显示长期记忆和语义索引数量，目前应为 0。

## 后续实现顺序

建议 v0.4 开始实现第二层长期记忆：

1. 主人命令手动添加、查看、删除长期记忆。
2. AI 根据对话自动提取“偏好、事实、性格特点、重要事件”。
3. 每次回复前读取当前用户或当前群的长期记忆。

建议 v0.5 再实现第三层语义索引：

1. 选择 embedding 模型。
2. 给长期记忆生成向量并写入 `memory_embeddings`。
3. 用户发言时计算查询向量，召回相似记忆。
4. 控制召回数量，避免旧记忆干扰当前对话。
