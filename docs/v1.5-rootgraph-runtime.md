# v1.5 RootGraph 统一运行入口设计

本文记录 v1.5 的目标、边界和设计方案；实际落地状态、QQ live 验证和补丁记录统一整理到 `docs/version-runlog.md`。

v1.5 的目标不是继续增加更多主人命令，也不是把所有旧命令一次性改写成 Agent 工具，而是搭建稳定的 RootGraph：让 QQ 事件进入系统后，先经过统一的权限、路由、上下文和提交边界，再分发到 ChatGraph、MainAgentGraph 等子图。

## 阶段定位

当前项目已经具备：

```text
普通聊天 ChatGraph
MemoryRAG 普通聊天语义记忆召回
ProjectDocRAG 开发侧项目文档召回
MainAgentGraph /agent 显式主人管理入口
ToolRegistry 工具注册和参数校验
ToolPolicyCheck 工具风险检查
agent_tasks / agent_approvals 审批记录和恢复执行
一批旧 QQ 主人管理命令的 /agent 语义化迁移
```

当前的问题不是单个能力缺失，而是入口层逐渐分散：

```text
NoneBot matcher 处理部分固定命令
普通聊天路径处理聊天上下文、记忆和回复
/agent 路径处理 MainAgent、工具和审批
权限检查、试用、黑名单、群白名单、是否响应群聊等逻辑仍散在多个入口附近
视觉、语音、记忆持久化、摘要压缩等运行期副作用没有统一生命周期视图
```

因此 v1.5 的核心目标是：

```text
建立 RootGraph 作为 QQ Runtime 的统一调度层。
```

RootGraph 不替代 ChatGraph 和 MainAgentGraph。RootGraph 只负责回答：

```text
这条事件是否应该处理？
它属于哪个会话和哪个用户角色？
应该路由到哪个能力图？
需要构建哪类上下文？
哪些副作用允许提交？
最终是否回复、回复什么、记录什么？
```

## 当前已有基础

当前代码里已经有 RootGraph 的第一层骨架：

```text
RuntimeState
RuntimeIntent
ActorContext
SessionContext
EventContext
RootGraphRunner
RouteDecision
runtime_state_from_main_agent_command
runtime_state_from_chat_request
```

当前 RootGraphRunner 已能按显式 `RuntimeIntent` 分发 handler，并记录基本 `root_graph` artifact。但它还不是完整运行入口：

```text
normalize_event 仍主要由 adapter 或上层调用方完成
load_actor_context 还没有收敛完整权限判断
hard_policy_gate 仍未统一接管黑名单、白名单、试用和群聊响应策略
route_intent 主要依赖显式 intent，没有完整从 QQ 事件自动判定
dispatch_capability 只完成最小 handler 分发
commit_side_effects 尚未作为 RootGraph 明确节点
错误、降级和观测还未形成统一规范
```

v1.5 就是在这个基础上把 RootGraph 从“显式分发器”推进为“运行入口骨架”。

## 设计原则

### 1. RootGraph 是交通枢纽，不是万能 Agent

RootGraph 不调用大模型做自由规划，不替代子图内部逻辑。

```text
RootGraph routes.
Subgraphs execute.
Policy gates decide.
```

具体能力继续由已有子图负责：

```text
ChatGraph：普通聊天表达和聊天运行
MainAgentGraph：/agent 工具请求、审批、开发上下文和主人管理
MemoryContextGraph：聊天记忆上下文
MemoryPersistGraph：聊天记忆持久化
MemoryRetrievalGraph：MemoryRAG 检索
VisionGraph：图片理解
VoiceGraph：语音输出
DiagnosticsGraph：诊断
MemoryAdminGraph：旧固定记忆管理命令
NotificationGraph：主人转告
```

### 2. 显式入口优先，普通聊天不误触工具

`/agent` 仍是 MainAgent 的明确入口。普通聊天不触发 ProjectDocRAG，不触发主人管理工具，不进入 ToolRegistry 写工具。

```text
/agent -> MainAgentGraph
普通聊天 -> ChatGraph
```

### 3. 权限由代码决定，不由提示词决定

以下判断必须在 RootGraph 或其前置 adapter 中由代码完成：

```text
主人识别
黑名单拦截
私聊白名单
群白名单
未知私聊试用
群聊是否 @
群聊自动回复评分是否命中
消息长度限制
冷却限制
MainAgent 是否允许当前会话
```

模型只能生成文本或结构化请求，不能决定自己是否有权限。

### 4. 上下文分层，不混用 RAG

RootGraph 要明确上下文通道：

```text
minimal_context：权限、事件、会话、角色，不加载重上下文
chat_context：普通聊天上下文，包括历史、摘要、长期记忆、MemoryRAG、图片描述、角色卡
dev_context：仅 /agent 显式开发侧查询使用 ProjectDocRAG
```

边界必须保持：

```text
MemoryRAG 可进入普通聊天。
ProjectDocRAG 只进入 /agent dev_context，不进入普通聊天。
```

### 5. 副作用集中提交

RootGraph 不应该让每个分支随意落库。v1.5 要把副作用分成两类：

```text
执行中副作用：
  LLM 调用
  embedding / RAG 检索
  视觉模型调用

提交副作用：
  写聊天消息
  写 agent_tasks / agent_approvals
  写动态访问控制 data/access.json
  消耗试用次数
  更新 TTS 候选
  触发摘要压缩
  记录错误和观测
```

第一版不一定把所有提交都迁到同一个函数，但设计上要把 `commit_side_effects` 作为 RootGraph 生命周期节点明确出来。

## 总体架构

目标结构：

```text
QQ / NapCat / OneBot
  -> NoneBot adapter
  -> RuntimeState
  -> RootGraph
       -> normalize_event
       -> load_actor_context
       -> hard_policy_gate
       -> route_intent
       -> build_runtime_context
       -> dispatch_capability
       -> commit_side_effects
       -> render_response
       -> observe_runtime
  -> QQ reply or silence
```

第一版可以保留部分 normalize 和 adapter 逻辑在外层，但 RootGraph artifact 必须能说明它实际做了什么。

## RuntimeState 设计

当前 `RuntimeState` 已有：

```text
event:
  message_id
  raw_text
  plain_text
  has_image

actor:
  user_id
  role

session:
  session_type
  session_key
  group_id

intent:
  RuntimeIntent | None

runtime:
  task_id
  artifacts
  tool_events
  response
  error
```

v1.5 建议在不大改结构的前提下，通过 `artifacts` 增加以下稳定区块：

```text
artifacts["policy"]:
  decision
  reason
  actor_role
  allow_chat
  allow_main_agent
  should_reply

artifacts["route"]:
  intent
  reason
  selected_handler

artifacts["context"]:
  context_level
  memory_rag_enabled
  project_doc_rag_enabled
  vision_used

artifacts["commit"]:
  user_message_saved
  bot_message_saved
  trial_updated
  compression_scheduled
  approval_created

artifacts["root_graph"]:
  node_trace
  route
  reason
  dispatched
  error
```

如果后续需要更强类型，再把这些 artifact 升级为 dataclass；MVP 阶段先用 dict，降低改动半径。

## RootGraph 节点

### normalize_event

目标：

```text
把 QQ 事件转成 RuntimeState。
提取纯文本、原始文本、图片标记、message_id。
确定 session_key：private:<user_id> 或 group:<group_id>。
```

第一版可继续复用现有 adapter：

```text
runtime_state_from_main_agent_command
runtime_state_from_chat_request
```

但 RootGraph artifact 中要记录 normalize 是否已完成。

### load_actor_context

目标：

```text
识别 owner / whitelisted / user / blocked。
读取 current_access() 合并后的动态和静态访问控制。
```

规则：

```text
bot_owner_qq 命中 -> owner
user_blacklist 命中 -> blocked
private_whitelist 命中 -> whitelisted
授权群成员在群聊中 -> whitelisted 或 user，具体按现有策略保守映射
其他 -> user
```

### hard_policy_gate

目标：

```text
在路由和模型调用前完成硬权限判断。
```

建议策略：

```text
blocked:
  denied 或 silence，不进入任何子图

owner + private + /agent:
  allow main_agent

owner + group + /agent:
  第一版仍按当前边界拒绝或不进入 MainAgent

non-owner + /agent:
  denied

private chat:
  owner 或 private whitelist 允许
  unknown private 按试用规则允许或拒绝

group chat:
  group whitelist 不命中则 ignore
  未 @ 且自动回复未命中则 ignore
  @ 或自动回复命中则 allow chat
```

`denied` 和 `ignore` 要分开：

```text
denied：可以回复一条拒绝原因，例如非主人调用 /agent。
ignore：静默，例如群聊未触发回复。
```

当前 `RuntimeIntent` 没有 `DENIED`，MVP 可用：

```text
state.error = "permission_denied"
state.response = 拒绝文案
RuntimeResponse(..., should_reply=True)
```

后续可再补 `RuntimeIntent.DENIED`。

### route_intent

路由优先级：

```text
1. 显式 /agent 或 /agent-debug -> MAIN_AGENT
2. 明确旧固定命令 -> 继续由 NoneBot matcher 兜底，不强行进 RootGraph
3. 普通私聊文本或图片 -> CHAT
4. 群聊 @ 或自动回复命中 -> CHAT
5. 其他群聊消息 -> IGNORE
```

第一版 RootGraph 不接管全部旧命令。旧命令迁移策略保持：

```text
/agent 语义工具：已迁移的一批主人管理命令走审批。
旧固定命令：继续由 matcher 直接兜底。
高风险旧命令：不放进 /agent 语义工具。
```

### build_runtime_context

目标：

```text
按 intent 选择上下文级别。
```

建议：

```text
MAIN_AGENT:
  minimal_context
  只在 dev_context 工具里使用 ProjectDocRAG

CHAT:
  chat_context
  可加载 MemoryRAG、短期历史、摘要、长期记忆、角色卡、图片描述

IGNORE / DENIED:
  minimal_context
```

第一版可以不把所有 ChatGraph 上下文加载搬进 RootGraph，只要 RootGraph 明确传入 ChatGraph 并记录 context_level。

### dispatch_capability

目标：

```text
按 route 调用子图 handler。
```

MVP handler：

```text
RuntimeIntent.MAIN_AGENT -> create_read_only_main_agent_runtime_handler(...)
RuntimeIntent.CHAT -> 现有聊天运行 handler 或 ChatGraph session adapter
RuntimeIntent.IGNORE -> no reply
```

要求：

```text
MainAgent 本地管理工具结果直接返回，不走角色卡语气总结。
ChatGraph 不接 ToolRegistry。
ProjectDocRAG 不进入 ChatGraph。
```

### commit_side_effects

目标：

```text
统一记录“本轮是否提交了副作用”。
```

第一版不要求把所有旧提交逻辑搬迁，但要在 artifact 中记录：

```text
chat 消息是否保存
bot 回复是否保存
试用次数是否消耗
是否创建 agent_task / approval
是否恢复执行审批工具
是否触发摘要压缩
```

后续可以逐步把分散在 ChatGraph / handler 中的提交动作收敛到 RootGraph commit 节点。

### render_response

目标：

```text
统一决定 should_reply 和 text。
```

规则：

```text
IGNORE -> should_reply=false
DENIED -> 根据场景决定是否回复；群聊尽量少刷屏
MAIN_AGENT -> 返回 MainAgent 文案
CHAT -> 返回 ChatGraph 文案
错误 -> 按错误类型降级
```

### observe_runtime

目标：

```text
给调试和后续 RAG 上下文恢复留下稳定线索。
```

RootGraph 每次执行至少记录：

```text
node_trace
actor_role
session_key
intent
policy_decision
route_reason
selected_handler
should_reply
error
```

不要求每次写文件；可以先写入 `state.artifacts["root_graph"]`，必要错误再走现有 error log。

## Intent 路由矩阵

第一版测试矩阵：

```text
owner 私聊 /agent 状态
  -> MAIN_AGENT
  -> should_reply=true

非 owner 私聊 /agent 状态
  -> denied
  -> 不调用 MainAgentGraph

owner 群聊 /agent 状态
  -> denied 或 group_denied
  -> 不调用 MainAgentGraph

owner 私聊普通消息
  -> CHAT

私聊白名单用户普通消息
  -> CHAT

未知私聊且试用可用
  -> CHAT
  -> commit 标记 trial_updated

未知私聊且试用不可用
  -> denied

黑名单用户普通消息
  -> denied 或 ignore
  -> 不调用 ChatGraph

授权群 @ 机器人
  -> CHAT

授权群未 @ 且自动回复不命中
  -> IGNORE
  -> should_reply=false

未授权群消息
  -> IGNORE 或 denied
  -> 不调用 ChatGraph

图片消息且允许聊天
  -> CHAT
  -> context 标记 vision_used 或 has_image

MemoryRAG 失败
  -> 降级 CHAT，不中断普通回复

MainAgent 工具错误
  -> 返回工具错误，不进入 ChatGraph 角色卡语气
```

## 和 MainAgentGraph 的关系

MainAgentGraph 已经完成 Route B 的关键部分：

```text
ToolRegistry
owner_read_command
owner_write_command
agent_task_read
agent_task_command
审批请求
审批确认和恢复执行
旧命令兜底
```

RootGraph 不重写这些能力，只负责：

```text
确保只有 owner private 能进 /agent。
确保 /agent 不被普通聊天吃掉。
确保 MainAgent 的结果不被 ChatGraph 或角色卡污染。
记录 MainAgent route / policy / dispatch artifact。
```

MainAgentGraph 内部仍负责：

```text
语义工具分类
ActionRequest 校验
ToolPolicyCheck
审批中断
审批恢复
工具执行
tool_result 渲染
```

## 和 ChatGraph 的关系

ChatGraph 继续负责普通聊天。

RootGraph 不应该把聊天 prompt 拼装细节全部吞掉。v1.5 只需要让 ChatGraph 明确成为 `CHAT` intent 的处理器。

ChatGraph 继续负责：

```text
短期历史读取
正式摘要注入
空窗摘要注入
手动长期记忆注入
MemoryRAG 注入
角色卡注入
图片描述上下文
调用聊天模型
保存聊天消息
更新 TTS 候选
触发摘要压缩
```

RootGraph 负责：

```text
是否允许进入 ChatGraph。
是否应该静默。
本轮上下文级别是什么。
ChatGraph 成功或失败后的统一响应和观测。
```

## 和旧固定命令的关系

v1.5 不做“一口吞掉所有旧命令”。

当前策略：

```text
已迁移到 /agent 语义工具的旧命令：
  继续走 /agent + 审批。

高风险旧命令：
  不进入 /agent 语义工具。
  继续保留旧固定命令兜底。

普通旧固定命令：
  暂时继续由 NoneBot matcher 处理。
```

不在 v1.5 迁移范围：

```text
/清空全部上下文
/清空全部摘要
/删除长期记忆
/重建记忆索引
/压缩当前会话
任意 shell
任意文件写入
任意数据库写入
```

## 错误和降级策略

RootGraph 第一版统一错误策略：

```text
permission_denied:
  私聊可以回复拒绝原因。
  群聊尽量静默或短回复。

group_ignored:
  should_reply=false。

main_agent_error:
  返回 MainAgent 错误，不进入 ChatGraph。

chat_llm_error:
  返回聊天失败提示。

vision_error:
  降级为无图聊天，必要时提示图片理解失败。

memory_rag_error:
  降级为普通聊天，不阻断回复。

commit_error:
  回复尽量保持用户可见结果，同时记录错误日志。
```

## 分阶段落地计划

### Phase 1：RootGraph 骨架补全

```text
补 RootGraph policy artifact。
补 route artifact。
让 RootGraph 明确支持 MAIN_AGENT / CHAT / IGNORE。
保持现有 /agent handler 和 ChatGraph handler 不大改。
补 RootGraph route matrix 单测。
```

### Phase 2：权限前置收敛

```text
把 owner / whitelist / blacklist / group allow / trial 的判定收敛成 policy_result。
减少 handler 内重复权限判断，但不急着删除旧保护。
旧保护保留为 defense in depth。
```

当前落地状态：

```text
已完成第一步：普通聊天会在进入 CHAT handler 前构建 chat_access_policy artifact。
RootGraph hard_policy_gate 会读取该 artifact，并在 allow_dispatch=false 时直接阻断 CHAT 分发。
已进入 artifact 的判断包括私聊/群聊开关、黑名单、白名单、群授权、陌生私聊试用次数、消息长度和普通聊天限速。
语义语音和其他旧路径仍保留 prepare_chat_request / check_access / check_message_limits 兜底。
```

### Phase 3：普通聊天接入 RootGraph

```text
普通私聊和授权群聊从 RootGraph CHAT intent 进入 ChatGraph。
确认 MemoryRAG 只在 ChatGraph 路径使用。
确认 ProjectDocRAG 仍只在 /agent dev_context 使用。
```

当前落地状态：

```text
已完成第一步：非语义语音的普通聊天入口先进入 RootGraph CHAT，再由 RootGraph handler 调用现有聊天运行链路。
本步保持现有 ChatGraph / legacy chat 分支、回复发送、持久化、压缩逻辑不变。
shadow_chat production_route 会记录 root_graph_chat。
RootGraph CHAT 返回 should_reply=false，避免 RootGraph 外层重复发送已经由聊天链路发送过的回复。
语义语音仍暂时走旧入口，后续在语音提交链路稳定后再纳入 RootGraph。
```

### Phase 4：提交和观测稳定

```text
统一记录 commit artifact。
统一记录 error artifact。
必要时补 RootGraph 最近观测命令。
```

当前落地状态：

```text
已完成第一步：聊天链路会写入 chat_commit artifact，RootGraph commit artifact 会汇总聊天提交状态。
当前可观测字段包括文本回复是否发送、语音是否发送、聊天轮次是否持久化、试用次数是否更新、压缩是否调度、TTS 候选是否更新、图片上下文是否 deferred、CHAT runtime stage。
已新增 `/agent RootGraph 最近观测` 只读入口，用于查看最近一次 RootGraph/CHAT 非正文观测快照。
该入口只输出会话、Actor、policy、route、context、commit、chat_runtime、chat_access_policy、chat_commit、shadow 摘要，不输出用户原文或回复正文。
本步不移动实际副作用执行位置，只让 RootGraph 能解释 handler 已完成哪些提交动作。
```

## 验收标准

v1.5 完成时应满足：

```text
RootGraph 是 /agent 和普通聊天的共同运行入口骨架。
```

## 非目标

v1.5 不做：

```text
多步自主 Agent loop
通用任务自动执行
任意 shell 工具
任意文件写入工具
任意数据库写入工具
自动长期记忆写入
自动角色卡修改
把全部旧命令迁入 RootGraph
把全部旧命令迁入 /agent 语义工具
清空全部摘要 / 清空全部上下文语义化
删除长期记忆语义化
异步审批恢复
```

## 风险点

### 1. 入口重复执行

如果 NoneBot matcher 和 RootGraph 同时处理同一条消息，可能重复回复。

策略：

```text
第一版明确哪些入口交给 RootGraph。
旧固定命令继续 block=True 兜底。
/agent 入口只走 MainAgent Runtime。
普通聊天入口再逐步迁入 RootGraph。
```

### 2. 群聊刷屏

RootGraph 错误处理如果不区分私聊和群聊，可能在群里刷错误。

策略：

```text
group ignore 默认 should_reply=false。
group denied 尽量短回复或静默。
LLM / RAG 错误在群聊里少说。
```

### 3. RAG 边界混淆

最重要边界：

```text
ProjectDocRAG 不进入普通聊天。
```

测试必须长期覆盖。

### 4. 权限收敛过快

如果一次性删除旧 handler 内权限判断，风险较大。

策略：

```text
RootGraph 前置权限是主防线。
旧 handler 权限保留一段时间作为防御层。
等 live 稳定后再考虑清理重复判断。
```

## 最小可交付版本

v1.5 最小可交付版本是：

```text
RootGraphRunner 支持 MAIN_AGENT / CHAT / IGNORE 的稳定分发。
RuntimeState artifact 记录 policy、route、root_graph。
/agent 继续走现有 MainAgentGraph。
普通聊天可通过 RootGraph 进入现有 ChatGraph handler。
权限和路由矩阵测试通过。
旧固定命令不受影响。
```

这个版本完成后，系统结构会从“多个入口各自处理”转向“RootGraph 统一调度，子图各司其职”。后续再做多步任务、复杂审批、通知外发或更细的观测，就会有更稳的承载面。
