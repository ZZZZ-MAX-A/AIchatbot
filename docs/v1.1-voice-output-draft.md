# v1.1 语音输出功能草案

## 状态

本文是语音输出功能的本地讨论草案和预研记录。当前功能已经进入 v1.1 初版实现，正式运行方式以 `docs/runbook.md` 为准。

历史预研阶段决定：

```text
暂不写机器人功能代码
暂不提交 GitHub
暂不作为 v1.0 功能
作为后续版本候选方向保留
本地 TTS 与 QQ 语音发送链路已经完成独立验证
```

当前 v1.1 实现状态：

```text
主人私聊语音触发已接入
上一条机器人回复转语音已接入
语义语音回复已接入
IndexTTS2 本地 TTS 服务已接入
本地 TTS 服务自动启动与 10 分钟空闲释放已接入
```

## 功能定位

语音输出不是诊断、记忆、摘要、转告一类的实用工具功能，而是角色表达和聊天气氛增强功能。

它的核心价值不是“把文字变成音频文件”，而是让角色在主人明确想听的时候，用语音说出一小段符合当前角色风格的话。

因此该功能应被定位为：

```text
主人专属的情绪表达增强功能
```

而不是：

```text
通用语音工具
默认语音聊天模式
群聊播报系统
文件生成系统
语音识别系统
```

## 不包含的能力

第一版不考虑以下能力：

```text
不做语音输入
不识别主人或其他用户发来的 QQ 语音
不要求机器人听懂语音
不让每一条回复都自动转语音
不做看图后自动语音回复
不让普通用户触发语音
不把诊断、转告、错误日志、配置状态转成语音
不批量朗读聊天记录
不长期保存生成的语音文件
```

## 推荐的第一版范围

第一版建议只做：

```text
主人私聊可用
自然语言触发
短文本转语音
上一条机器人回复转语音
角色生成短句后转语音
发送失败时降级为文字说明
语音文件临时保存并自动清理
```

群聊建议默认关闭。后续如果要支持群聊，也应只允许主人在明确场景中触发，并加入限流。

第一版不让聊天 AI 自主决定是否发语音。是否触发语音由语音模块根据权限、场景、动作句式和安全规则判断。

第一版的“上一条机器人回复”不从数据库里临时模糊查找，而是由运行时显式维护最近一条可朗读普通聊天回复。

```text
last_tts_candidate = 主人私聊中，机器人最近一次发送成功的可朗读普通聊天回复
```

只有普通聊天模型生成的自然文本回复可以更新 `last_tts_candidate`。

不进入 `last_tts_candidate` 的内容：

```text
命令执行结果
诊断结果
状态信息
摘要查看结果
转告结果
错误提示
语音模块自己的成功或失败提示
过长回复
敏感内容
空回复
纯配置或大段代码
```

机器人重启后，第一版可以丢失 `last_tts_candidate`。语音输出是临时表达增强功能，不承担历史回放职责。

## 触发方式

语音输出不应主要依赖 `/生成语音` 这类命令。

原因：

```text
斜杠命令更像操作机器人
自然语言更像和角色对话
语音功能本身偏气氛和情绪价值
频繁使用命令会削弱代入感
```

推荐用自然语言触发，例如：

```text
用语音说给我听
用声音跟我说晚安
这句念给我听
刚刚那句话可以语音吗
把上一条读出来
把“我一直在这里”念出来
这段话用语音发我
想听你说一句鼓励我的话
```

触发条件应识别的是“生成语音动作”，不是简单匹配“声音”“语音”等词。

不应触发的例子：

```text
你觉得语音功能怎么样
这个角色的声音是什么样的
以后能不能加语音
动漫角色的配音很好听
```

## 三类语音请求

### 1. 朗读指定内容

用户明确给出要朗读的文本。

示例：

```text
主人：用语音说：晚安，今天也辛苦了。
机器人：发送语音“晚安，今天也辛苦了。”
```

这是最清晰、最安全的触发类型。

### 2. 朗读上一条机器人普通回复

用户先看到文字回复，再决定是否转语音。

示例：

```text
主人：帮我写一句温柔一点的晚安
机器人：晚安，今天辛苦了。剩下的事情明天再说。

主人：刚刚那句用语音说给我听
机器人：发送上一条回复的语音版
```

这种方式有较好的体验，因为主人可以先确认文本是否合适，再选择转语音。

这里的“上一条回复”仅指 `last_tts_candidate`，不等于数据库中的任意上一条机器人消息。

如果不存在可朗读候选，推荐回复：

```text
我现在没有可朗读的上一条回复。
```

### 3. 生成一小句气氛语音

用户没有指定完整文本，而是提出情绪表达需求。

示例：

```text
主人：用语音跟我说句晚安
机器人：生成符合角色卡风格的短句，并发送语音
```

这一类最有角色感，但也最需要长度限制和安全边界。第一版应只允许生成一两句短文本，不允许生成长篇语音。

## 角色卡关系

语音输出应和角色卡协作，但不能让角色卡控制底层权限。

推荐分层：

```text
角色卡：影响文本内容、称呼、语气和表达风格
语音模块：决定是否允许生成语音、朗读哪段文本、长度是否合规
安全规则：决定是否拒绝敏感内容、是否允许当前场景、是否允许当前用户
TTS 引擎：只负责把允许朗读的文本转成音频
```

角色卡可以影响“说什么”，但不能决定：

```text
是否绕过主人限制
是否在群聊发语音
是否朗读敏感信息
是否突破长度限制
是否读取本地文件
是否朗读聊天数据库
是否自动把所有回复都转语音
```

## 与记忆和摘要的关系

语音请求通常不应进入长期记忆。

例如：

```text
用语音说句晚安
这句念给我听
刚刚那句话转语音
```

这些一般是一时的表达请求，没有长期记忆价值。

除非主人明确说：

```text
以后我喜欢你偶尔用语音说晚安，记住这个偏好。
```

否则不应把语音请求写入长期记忆或摘要。

摘要系统应继续关注：

```text
主人明确偏好
项目决策
系统设置
长期约束
待办事项
重要纠正
```

不应被大量“生成语音”“念给我听”类事件污染。

## 与转告功能的关系

语音输出不应参与主人转告流程。

转告功能是严肃、固定模板、不可改写的消息传递功能。语音输出是情绪表达功能，两者边界不同。

不推荐：

```text
普通用户转告主人后，机器人把转告内容生成语音
机器人用语音播报转告内容
机器人把转告内容交给角色卡润色后语音发送
```

转告仍应保持文字、固定格式、可审计。

## 与视觉功能的关系

第一版不做看图后自动语音回复。

原因：

```text
视觉识别已经有安全边界
图片内容可能包含隐私
看图回复语音的使用频率不高
语音输出会增加误触发和打扰
```

后续如果要支持，也应保持明确触发，例如：

```text
主人先让机器人看图
机器人文字回复
主人再说：刚刚那句用语音说
```

## 权限建议

第一版建议：

```text
只给主人开放
优先只在私聊开放
群聊默认关闭
普通用户不能触发
管理命令不能通过语音自然请求执行
```

如果后续支持群聊，也应满足：

```text
必须是主人触发
必须明确要求发语音
必须限流
必须限制文本长度
默认不响应普通群友的语音请求
```

## 长度和频率限制

语音输出应短而克制。

建议第一版限制：

```text
单条语音 80 字以内
最多不超过 150 字
超过长度则拒绝或要求主人缩短
私聊也需要冷却时间
群聊如果开放，需要更严格限流
```

不建议自动截断长文本。自动截断可能让主人误以为完整内容已经生成。

推荐提示：

```text
这段太长了，当前语音最多支持 150 字。你可以让我读其中一句。
```

## 安全边界

即使只做语音输出，也必须防止敏感内容被朗读或外发。

应拒绝或脱敏的内容包括：

```text
API Key
Token
密码
验证码
手机号
身份证号
银行卡号
数据库内容
.env 配置
本地文件路径中的敏感信息
其他用户隐私
批量聊天记录
角色卡完整内容
```

语音功能不能成为读取本地文件、导出数据库、泄露配置的入口。

## 文件处理

TTS 会生成本地音频文件，因此需要临时文件管理。

建议：

```text
使用固定临时目录
文件名使用随机 ID 或时间戳
不要把用户文本作为文件名
发送成功后清理
发送失败后也进入定期清理
不长期保存语音文件
不把音频路径写入聊天记忆
```

示例路径：

```text
D:\AIchatbot\temp_audio\tts-20260624-223000-a8f3.wav
```

不推荐：

```text
D:\AIchatbot\audio\我的手机号是138xxxx.wav
```

## TTS 引擎选择

当前决定：v1.1 第一版不使用远程 TTS，优先使用本地 TTS。

当前第一候选：

```text
IndexTTS2 local
```

原因：

```text
本地运行，不把待朗读文本发送给远程 TTS 服务
中文短句效果更符合角色表达方向
支持音色参考和情绪控制
输出 WAV 可被 NapCat record 发送
```

重要边界：

```text
聊天模型负责生成最终文本
IndexTTS2 只负责朗读最终文本
TTS 不读取角色卡全文
TTS 不改写文本
TTS 不判断权限
TTS 不读取聊天数据库
TTS 不参与转告、诊断、摘要或记忆
```

IndexTTS2 不应直接嵌入 NoneBot 主进程。推荐作为独立本地服务常驻运行：

```text
QQ 机器人主进程：
权限、触发、文本选择、长度限制、敏感过滤、调用 TTS、发送 QQ、清理文件

IndexTTS2 本地服务：
加载模型、接收最终文本、生成 WAV、返回临时音频路径
```

推荐服务边界：

```json
{
  "text": "晚安，今天辛苦了。剩下的事情明天再说。",
  "voice": "default",
  "emotion": "calm",
  "output_format": "wav"
}
```

返回：

```json
{
  "ok": true,
  "audio_path": "D:\\AIchatbot\\temp_audio\\tts-20260625-160000-a8f3.wav",
  "duration_ms": 4300
}
```

后续仍可保留 provider 抽象，以便替换为其他本地 TTS：

```text
TTS_PROVIDER=indextts2_local
TTS_PROVIDER=piper_local
TTS_PROVIDER=chattts_local
```

远程 TTS 暂不作为第一版方案。

## TTS 引擎取舍

本地 TTS 优点：

```text
隐私更好
长期成本低
不依赖外部 API
更适合长期维护
```

本地 TTS 缺点：

```text
安装和部署复杂
可能需要额外模型
Windows 音频依赖可能麻烦
音质和速度需要实测
```

云端 TTS 优点：

```text
接入简单
音质通常较好
可选音色多
速度较稳定
```

云端 TTS 缺点：

```text
可能收费
需要联网
文本会发送给第三方
API 变更会影响稳定性
```

建议后续不要把实现写死到单一服务，而是预留配置：

```env
ENABLE_TTS=false
TTS_PROVIDER=indextts2_local
TTS_VOICE=default
TTS_STYLE=calm
TTS_SPEED=1.0
TTS_MAX_CHARS=150
TTS_PRIVATE_ONLY=true
TTS_GROUP_ENABLED=false
TTS_COOLDOWN_SECONDS=60
TTS_OUTPUT_FORMAT=wav
TTS_TEMP_DIR=temp_audio
TTS_TIMEOUT_SECONDS=30
```

## QQ 发送方式

需要实际验证 NapCat / OneBot 对语音发送的支持。

可能路线：

```text
发送 QQ 语音消息
发送音频文件
生成失败时只回复文字
```

优先目标是发送 QQ 语音消息。但如果格式兼容不稳定，可以先退而求其次发送音频文件。

需要验证：

```text
NapCat 是否支持发送 record
本地文件路径是否可用
支持 wav、mp3、silk、amr 中的哪一种
是否需要 ffmpeg 或 silk 转码
私聊和群聊行为是否一致
发送失败时错误如何返回
```

仍待验证：

```text
mp3 是否可直接发送
```

```text
TTS 生成短 WAV

失败时降级为文字提示
发送成功或失败后清理临时音频文件
```

## 管理命令


## 推荐交互

### 指定文本

```text
主人：用语音说：晚安，今天也辛苦了。
机器人：发送短语音
```

### 上一条回复

```text
主人：刚刚那句念给我听
机器人：发送上一条机器人回复的语音版
```

### 气氛短句

```text
主人：用语音跟我说句晚安
机器人：根据角色卡生成一句短句并发送语音
```

## 不推荐交互

```text
每条机器人回复都自动发语音
普通群友触发语音
长文自动朗读
看图后自动语音回复
诊断结果语音播报
转告内容语音播报
自动情绪检测后主动发语音
角色卡自行决定绕过限制发语音
```

## 后续版本建议

如果进入开发，建议分阶段：

```text
第一阶段：主人私聊自然语言触发，短文本 TTS，上一条机器人普通回复转语音，发送失败降级文字
第二阶段：增加语音状态、临时文件清理诊断、本地 TTS 服务健康检查
第三阶段：评估是否允许主人在指定群聊中低频触发
第四阶段：再考虑多音色、语速配置或更细的角色语音风格
```

不建议第一版加入主动情绪识别后自动发语音。这个能力更容易误触发，也更难维护，应等基础语音输出稳定后再讨论。

## 当前结论

语音输出功能具备可行性，也符合角色聊天机器人的情绪表达方向，但它不是 v1.0 稳定性的必需功能。

如果后续实现，应遵循：

```text
主人专属
自然触发
短句优先
低频克制
不影响正常文字聊天
不参与诊断、转告、记忆和摘要
不让角色卡控制权限
失败时安全降级
本地 TTS 优先
TTS 服务独立于机器人主进程
```

该功能应作为角色表达增强模块，而不是通用工具模块。
