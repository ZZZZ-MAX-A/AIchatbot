# 让 QQ AI 机器人“会说话，也会接住上下文”：语音输出与记忆运行结构设计

> 本文基于项目 v1.1 和 v1.2 的实现记录整理，重点介绍 QQ AI 机器人如何接入本地语音输出，以及如何重新梳理短期原文、正式会话摘要、空窗场景摘要和手动长期记忆的运行结构。一个偏表达体验，一个偏上下文工程，但它们都服务于同一个目标：让机器人在长期聊天中更自然、更稳定、更不容易丢上下文。

## 摘要 / 导语

在基础聊天、权限、摘要、角色卡、转告、识图和诊断都跑通之后，项目进入了两个更细的方向：

```text
v1.1：语音输出，让机器人在主人明确触发时发送 QQ 语音
v1.2：记忆系统运行结构，让上下文在长对话里更稳定
```

这两个方向看起来差别很大。

语音输出更像体验增强：它让角色在合适的时候不只发文字，也能用本地 TTS 生成一段 QQ 语音。

记忆运行结构更像底层工程：它解决最近 40 条原文之外、正式压缩之前，中间那段上下文容易断掉的问题。

但这两个版本的共同点是：

```text
都不能让 AI 自由发挥底层权限。
```

语音不能让角色卡决定什么时候发；记忆不能让大模型自动把闲聊写成长期事实。真正稳定的 AI 机器人，必须把表达、权限、记忆和工具边界拆开。

## 建议标签

`QQ机器人`、`NoneBot2`、`NapCatQQ`、`OneBot`、`AI应用开发`、`TTS`、`上下文工程`、`Python`

## 当前系统背景

项目的主链路仍然是：

```text
QQ
  -> NapCatQQ
  -> OneBot v11
  -> NoneBot2
  -> AIchatbot plugin
  -> 聊天模型
  -> QQ 回复
```

语音输出增加了一条本地服务链路：

```text
聊天文本
  -> 语音触发和权限判断
  -> 文本清理和长度限制
  -> 本地 IndexTTS2 服务
  -> WAV 文件
  -> OneBot record 消息段
  -> QQ 私聊语音
```

记忆系统则围绕数据库和 prompt 上下文运行：

```text
messages 原文
session_summaries 正式摘要
gap_scene_summaries 空窗场景摘要
long_term_memories 主人手动长期记忆
```

一个是“怎么说出来”，一个是“说之前要参考什么”。

## v1.1：语音输出的定位

v1.1 的语音输出不是通用语音工具。

它不是：

```text
语音输入
语音识别
群聊播报系统
每条回复自动转语音
文件生成工具
诊断结果朗读器
转告内容播报器
```

它的定位更窄：

```text
主人私聊中的角色表达增强功能。
```

也就是说，只有在主人明确想听的时候，机器人才把短句转成语音。

这条边界很重要。

语音比文字更打扰人，也更容易产生“机器人过度主动”的感觉。第一版如果直接做成“每条消息都自动语音回复”，体验很可能很快失控。

## 三类语音请求

第一版语音输出主要覆盖三类请求。

### 1. 朗读指定文本

用户明确给出要朗读的内容：

```text
用语音说：晚安，今天也辛苦了。
```

机器人不需要调用聊天模型生成语义，只需要把冒号后的短文本转语音。

这是最安全的一类，因为待朗读文本来自主人明确指定。

### 2. 朗读上一条机器人回复

用户先看到文字，再决定是否转语音：

```text
主人：帮我写一句晚安
机器人：晚安，今天辛苦了。剩下的事情明天再说。

主人：刚刚那句念给我听
机器人：发送上一条回复的语音版
```

这里的“上一条回复”不是从数据库里模糊搜索，而是运行时维护一个候选：

```text
last_tts_candidate
```

只有普通聊天模型生成的、适合朗读的短文本，才会进入候选。

不进入候选的内容包括：

```text
诊断结果
配置状态
摘要查看结果
转告结果
错误提示
语音模块自己的提示
过长回复
敏感内容
大段代码
```

这样能避免用户一句“上一条念给我听”，机器人把诊断日志或配置说明读出来。

### 3. 语义语音回复

用户没有给完整文本，而是提出一个表达需求：

```text
用语音跟我说句晚安
用语音安慰我一下
```

这时聊天模型先根据角色卡生成短句，然后 TTS 负责朗读。

这一类最有角色感，但也最需要限制：

```text
只允许主人
优先只在私聊
短句优先
不读敏感内容
不生成长篇语音
失败时降级文字说明
```

第一版不让聊天 AI 自己决定“我现在要发语音”。是否触发语音由语音模块判断。

## 为什么不用斜杠命令作为主要入口

语音功能偏聊天体验，不像 `/诊断`、`/配置状态` 这类运维命令。

如果主要入口是：

```text
/生成语音 晚安
```

会显得很工具化。

更自然的触发方式是：

```text
用语音说给我听
刚刚那句念给我听
用声音跟我说晚安
这段话用语音发我
```

但自然语言触发不能等于关键词乱匹配。

下面这些不应该触发语音：

```text
你觉得语音功能怎么样
这个角色的声音是什么样的
以后能不能加语音
动漫角色的配音很好听
```

语音模块识别的是“生成语音动作”，不是简单看到“语音”“声音”就触发。

## 语音和角色卡的边界

角色卡可以影响“说什么”，不能决定“能不能发语音”。

推荐分层：

```text
角色卡：影响文本内容、称呼、语气和表达风格
语音模块：决定是否允许生成语音、朗读哪段文本、长度是否合规
安全规则：决定是否拒绝敏感内容、是否允许当前场景和用户
TTS 引擎：只负责把允许朗读的文本转成音频
```

角色卡不能控制：

```text
是否绕过主人限制
是否在群聊发语音
是否朗读敏感信息
是否突破长度限制
是否读取本地文件
是否朗读聊天数据库
是否自动把所有回复都转语音
```

这和前面几版的设计是一致的：表达规则不能覆盖权限规则。

## 与诊断、转告、视觉和记忆的关系

语音输出不应该参与诊断。

例如：

```text
/诊断
/配置状态
/最近错误
```

这些输出应该保持文字、固定格式、可复制、可排查，不应该变成语音。

语音输出也不应该参与转告。

转告是固定模板、保真、可审计的消息传递，不能被语音模块改写或朗读。

视觉功能也不自动触发语音。

更稳的流程是：

```text
主人让机器人看图
机器人文字回复
主人再说：刚刚那句用语音说
```

语音请求本身通常也不应该进入长期记忆。

例如：

```text
用语音说句晚安
刚刚那句念给我听
```

这类请求是临时交互，不是长期偏好。除非主人明确说：

```text
以后我喜欢你偶尔用语音说晚安，记住这个偏好。
```

否则不应该自动写入长期记忆。

## 本地 TTS：为什么选择 IndexTTS2 服务化

第一版选择本地 TTS，而不是远程 TTS。

主要原因：

```text
待朗读文本不发送给第三方 TTS
长期成本更可控
中文短句效果适合角色表达
输出 WAV 可通过 NapCat record 发送
```

当前候选是：

```text
IndexTTS2 local
```

但 IndexTTS2 不适合直接塞进 NoneBot 主进程。

原因是：

```text
模型加载慢
显存占用高
依赖环境复杂
主机器人进程不应该被 TTS 推理阻塞
```

所以更合理的结构是：

```text
QQ 机器人主进程：
权限、触发、文本选择、长度限制、敏感过滤、调用 TTS、发送 QQ、清理文件

IndexTTS2 本地服务：
加载模型、接收最终文本、生成 WAV、返回临时音频路径
```

TTS 服务只处理最终文本，不读取角色卡，不访问数据库，不判断权限。

## 从 WAV 到 QQ 语音

NapCat / OneBot 支持 `record` 消息段。

发送 payload 形态类似：

```json
{
  "action": "send_private_msg",
  "params": {
    "user_id": "主人 QQ",
    "message": [
      {
        "type": "record",
        "data": {
          "file": "D:\\AIchatbot\\temp_audio\\tts_xxx.wav"
        }
      }
    ]
  }
}
```

项目已经验证过：

```text
本地 WAV 文件可以通过 OneBot record 消息段发送到 QQ 私聊
IndexTTS2 生成的 WAV 可以被 NapCat 发送为 QQ 语音消息
第一版不需要远程 TTS
第一版暂不需要 silk/amr 转码
```

这条链路跑通后，语音功能的难点就从“能不能发出去”转成了“什么时候该发、发什么、失败时怎么降级”。

## 语音服务的运行风险

本地 TTS 最大的问题是慢，尤其是冷启动和显存紧张时。

因此需要几个运行策略：

```text
TTS 服务可手动启动，也可自动启动
模型空闲一段时间后释放显存
请求有超时
失败时降级为文字说明
临时音频文件定期清理
```

配置可以类似：

```env
ENABLE_TTS=true
TTS_SERVICE_URL=http://127.0.0.1:7861
TTS_MAX_CHARS=180
TTS_MAX_TOTAL_SECONDS=60
TTS_AUTO_START=true
TTS_STARTUP_WAIT_SECONDS=45
TTS_TIMEOUT_SECONDS=180
```

如果偶发超时，但服务端最终生成了 WAV，通常说明链路不是断的，更可能是显存、冷启动或服务状态问题。

排查时不要急着改聊天代码，应该先看：

```text
TTS 服务 /health
temp_audio 是否生成有效 wav
显存占用
是否刚启动
客户端超时配置是否过短
```

## 语音文本清理

聊天回复不一定适合直接朗读。

语音前需要做轻量清理，例如：

```text
删除括号和星号动作描写
删除明显不适合朗读的格式
按完整句子分段
段间加入短暂停顿
限制总字数和总时长
```

例如，文字回复里可能有：

```text
（轻轻点头）晚安，今天辛苦了。
```

语音里更适合读：

```text
晚安，今天辛苦了。
```

这里的目标不是重新创作，而是把已经允许朗读的文本变得更适合 TTS。

## v1.2：记忆运行结构为什么要重做

v1.2 的重点是记忆系统运行结构。

前面已经有：

```text
短期原文 messages
正式会话摘要 session_summaries
主人手动长期记忆 long_term_memories
```

但长对话里会出现一个空窗问题。

假设每次只给模型最近 40 条原文：

```text
MAX_CONTEXT_MESSAGES=40
```

当一个会话已经有 80 条未压缩消息时，最近 40 条能进上下文，但第 1-40 条可能已经不在短期窗口里，同时又还没触发正式压缩。

于是中间这段内容会暂时掉出上下文。

v1.2 引入“空窗场景摘要”，就是为了解决这个问题。

## 当前上下文层级

v1.2 的上下文大致按下面几层运行：

```text
系统规则
角色卡
手动长期记忆
最近正式会话摘要
空窗场景状态摘要
最近 40 条原文 messages
当前用户消息
```

各层职责不同：

```text
系统规则：权限、安全和底层行为边界
角色卡：表达方式
手动长期记忆：主人明确要求长期保存的事实/偏好
正式会话摘要：较早阶段聊天的大意
空窗场景摘要：还没正式压缩、也不在最近 40 条里的中间上下文
最近原文：当前细节
当前用户消息：最高优先级的即时输入
```

这套结构避免把所有东西都塞进一个大 prompt。

## 关键参数

推荐配置：

```text
MAX_CONTEXT_MESSAGES=40
ENABLE_MEMORY_COMPRESSION=true
MAX_STORED_MESSAGES_PER_SESSION=120
SUMMARY_KEEP_RECENT_MESSAGES=40
SUMMARY_BATCH_MESSAGES=80
SUMMARY_MIN_SOURCE_MESSAGES=40
MAX_SESSION_SUMMARIES_IN_CONTEXT=3

ENABLE_GAP_SCENE_SUMMARIES=true
GAP_SCENE_SUMMARY_1_THRESHOLD=40
GAP_SCENE_SUMMARY_2_THRESHOLD=80
MAX_GAP_SCENE_SUMMARIES_IN_CONTEXT=2

ENABLE_LONG_TERM_MEMORY_CONTEXT=true
MAX_LONG_TERM_MEMORIES_IN_CONTEXT=8
```

核心关系是：

```text
120 = 80 + 40
```

含义：

```text
未压缩原文超过 120 条后触发正式压缩
最旧 80 条压缩成正式会话摘要
最近 40 条继续保留为原文
正式压缩后清理已覆盖的空窗场景摘要
```

这个结构让上下文有明确生命周期，而不是无限堆积。

## 运行周期

设当前会话未压缩原文数量为 `N`。

### N <= 40

没有空窗。

```text
[最近 N 条原文]
```

此时模型能看到所有未压缩细节，不需要额外摘要。

### 40 < N <= 80

最近 40 条进入上下文，前面的 `N - 40` 条已经离开短期窗口。

此时生成第 1 条空窗场景摘要：

```text
[空窗摘要 1]
[最近 40 条原文]
```

它用于桥接刚离开短期窗口但还没正式压缩的内容。

### 80 < N <= 120

最近 40 条进入上下文，前面最多 80 条形成空窗。

此时生成第 2 条空窗场景摘要：

```text
[空窗摘要 1]
[空窗摘要 2]
[最近 40 条原文]
```

当 `N = 120` 时，覆盖关系是：

```text
空窗摘要 1：覆盖第 1-40 条未压缩原文
空窗摘要 2：覆盖第 41-80 条未压缩原文
最近原文：第 81-120 条
```

### N > 120

触发正式压缩：

```text
第 1-80 条 -> 正式会话摘要
第 81-120 条 -> 保留原文
空窗摘要 1、2 -> 清理
```

正式摘要是长期一点的会话摘要，空窗摘要只是临时桥接层。

## 空窗场景摘要不是长期记忆

空窗场景摘要很容易被误用。

它不是长期记忆，也不是角色卡规则，更不是用户画像。

固定字段可以设计为：

```text
场景位置：
场景类型：
场景分析：
已明确发生：
当前话题方向：
```

允许记录：

```text
明确设定的场景位置
当前对话类型
当前互动氛围
已明确发生或明确确认的事实
后续需要接住的话题方向
```

禁止记录：

```text
长期回复风格
2-3 句偏好
括号动作描写偏好
称呼偏好
主人心理诊断
AI 自己的情绪
未明确发生的亲密行为
API Key、Token、密码、二维码、身份证、手机号等敏感内容
```

尤其要注意，场景位置只记录明确设定，不能根据“困了”“洗澡”等词自动脑补地点。

摘要系统最怕编造。宁可写“未明确”，也不要替用户补剧情。

## 手动长期记忆仍然由主人控制

v1.2 没有让 AI 自动写长期记忆。

系统不会：

```text
自动从对话中提取长期事实
自动从对话中提取主人偏好
自动把旧会话摘要迁移到长期记忆
自动修改、删除或合并长期记忆
```

长期记忆仍然只支持主人手动维护：

```text
/添加事实记忆 内容
/添加偏好记忆 内容
/查看长期记忆
/删除长期记忆 记忆ID
```

长期记忆分两类：

```text
事实摘要：稳定事实，例如项目背景、群聊事实、当前系统长期设定
偏好摘要：主人长期偏好，例如协作方式、回复风格、明确要求长期生效的习惯
```

这和空窗摘要、正式会话摘要的职责不同。

如果所有摘要都能自动变成长期记忆，系统很快会污染。

## 记忆写入和读取的边界

运行时需要区分两件事：

```text
聊天前：构建 prompt 上下文
聊天后：保存消息并调度压缩
```

聊天前需要做：

```text
读取手动长期记忆
读取最近正式会话摘要
检查/更新空窗场景摘要
读取最近原文
组装上下文
```

聊天后需要做：

```text
保存用户消息
保存助手消息
检查是否需要正式压缩
更新 TTS 候选
```

后续 LangGraph 化时，这两段可以拆成两个 runner：

```text
MemoryContextGraphRunner
MemoryPersistGraphRunner
```

一个负责 prompt 前的上下文，一个负责回复后的持久化。

这样比在聊天函数里散落一堆记忆调用更清晰，也更容易测试。

## 调试命令

记忆系统必须可观察。

常用命令：

```text
/记忆状态
/摘要状态
/查看摘要
/查看空窗摘要
/查看长期记忆
```

`/摘要状态` 应该能看到：

```text
当前未摘要消息数
自动压缩阈值
保留最近原文数量
每次压缩条数
上下文摘要数
当前空窗摘要数
空窗摘要阈值
```

`/查看空窗摘要` 应该能看到：

```text
slot
覆盖消息数量
覆盖消息 ID 范围
更新时间
摘要正文
```

这些命令不只是给开发者看的，也能帮助主人判断机器人为什么“记得”或“不记得”某些上下文。

## 语音和记忆放在一起后的注意事项

语音功能接入后，很容易污染记忆。

例如：

```text
用语音说晚安
刚刚那句念给我听
这段话用声音发我
```

这些大多数只是临时表达请求，不应该频繁进入长期记忆或摘要重点。

摘要系统应该优先保存：

```text
项目决策
明确偏好
长期约束
重要纠正
待办事项
验收结果
```

而不是记录大量：

```text
主人要求机器人生成语音
机器人发送了语音
主人想听一句晚安
```

当然，如果语音请求本身代表长期偏好，主人应该明确添加：

```text
/添加偏好记忆 主人喜欢在晚安场景下偶尔使用短语音。
```

这就是“临时互动”和“长期偏好”的边界。

## 实测和验证思路

语音输出测试：

```text
主人私聊发送“用语音说：晚安”
机器人发送 QQ 语音
主人让机器人朗读上一条普通回复
不存在可朗读候选时给出文字提示
语义语音回复能先生成短文本再发语音
诊断结果、摘要查看、转告结果不进入语音候选
TTS 服务不可用时安全降级
生成的临时 WAV 能被清理
```

记忆运行测试：

```text
N <= 40 时只使用最近原文
40 < N <= 80 时生成第 1 条空窗摘要
80 < N <= 120 时生成第 2 条空窗摘要
N > 120 时正式压缩旧 80 条
正式压缩后清理已覆盖空窗摘要
/摘要状态 能显示未摘要消息数和阈值
/查看空窗摘要 能看到覆盖范围
手动长期记忆能注入上下文
AI 不自动写长期记忆
```

QQ 侧实测确认项：

```text
普通聊天记忆正常
/记忆状态 正常
/摘要状态 正常
语音 record 能正常发送到主人私聊
```

这些测试不一定都需要真实模型参与。很多边界可以通过纯本地单元测试覆盖，真实 QQ 只验证接入链路和用户体验。

## 小结

v1.1 和 v1.2 解决的是两个很实际的问题。

语音输出让机器人在主人明确触发时，可以用本地 TTS 发出短语音：

```text
主人专属
自然触发
短句优先
本地 TTS
失败降级
不参与诊断、转告和记忆管理
```

记忆运行结构让长对话里的上下文更稳定：

```text
最近 40 条保留原文
较早内容进入正式会话摘要
中间空窗由临时场景摘要桥接
长期记忆只由主人手动维护
角色卡仍只控制表达方式
```

这两个版本都不是单纯“加功能”。

它们真正做的是继续拆清楚边界：

```text
语音是表达，不是权限
角色卡是风格，不是控制器
摘要是上下文，不是长期画像
长期记忆是主人明确维护的事实和偏好
TTS 是工具，不参与决策
```

对一个长期在线的 QQ AI 机器人来说，这种边界比功能数量更重要。

因为系统越复杂，越需要知道每个模块该做什么、不该做什么，以及出了问题时应该从哪里查。
