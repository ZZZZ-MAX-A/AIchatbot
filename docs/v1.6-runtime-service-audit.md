# v1.6 Runtime service 解耦审计

本文记录 P2.1-P2.4 后的 Runtime service 解耦状态、QQ live 回归点，以及 `src/plugins/ai_chat/__init__.py` 后续是否继续拆分的建议。

当前日期：2026-07-07。

## 当前结论

MainAgent owner 主线已经完成代码层 service 解耦：

```text
owner_agent_runtime.py
  任务、审批、工作台、详情卡、审批恢复入口。

owner_read_runtime.py
  主人只读管理命令分发。

owner_write_runtime.py
  已审批主人写命令执行。

owner_runtime_factory.py
  task/read/write 三块 owner runtime 总装层。
```

`src/plugins/ai_chat/__init__.py` 仍然是 NoneBot/QQ adapter，但它不再直接承载 MainAgent owner 的 task/read/write runtime 细节。

本轮解耦不是独立进程、不是 HTTP API、不是 Web Owner Console，也没有改变数据库 schema 或新增执行能力。

## 已完成的 owner runtime 分层

```text
__init__.py
  QQ / NoneBot adapter
  事件接入、命令注册、owner 鉴权、RootGraph 调度、回复发送、依赖绑定。

owner_runtime_factory.py
  MainAgent owner runtime 总装层
  统一组装 owner_agent_runtime、owner_read_runtime、owner_write_runtime。

owner_agent_runtime.py
  任务和审批 runtime
  使用 OwnerAgentContext(session_key, user_id)，不依赖 QQ MessageEvent。

owner_read_runtime.py
  owner_read_command 只读命令 runtime
  只做只读命令分发，读取能力通过 OwnerReadRuntime 注入。

owner_write_runtime.py
  owner_write_command 已审批写 runtime
  只执行已注册、审批恢复允许的主人管理写命令。
```

## 行为边界

仍然保持：

```text
普通聊天不会触发 MainAgent owner runtime。
/agent 仍是主人私聊显式入口。
ProjectDocRAG 只在 /agent 显式 dev_context 中使用。
owner_read_command 只读，不写数据库、不发额外 QQ 消息。
owner_write_command 必须先审批。
确认审批后只恢复已注册且 approval_resume_enabled=true 的工具。
不执行 shell。
不执行任意文件写入。
不执行未注册数据库写入。
不开放多步写执行能力。
不新增 Web Owner Console。
不新增 HTTP API。
不改 DB schema。
```

## `__init__.py` 剩余职责审计

当前 `src/plugins/ai_chat/__init__.py` 约 5100 行。MainAgent owner 主线已拆出，但还有以下大块职责。

### 1. QQ adapter 必须保留

建议保留在 `__init__.py`：

```text
NoneBot command / matcher 注册。
MessageEvent / GroupMessageEvent / PrivateMessageEvent 适配。
matcher.finish / bot.send 这类 QQ 回复发送。
命令别名和 handler 入口。
QQ 权限前置检查。
RootGraph 顶层事件接入。
```

这些逻辑天然属于 adapter，没必要为了追求文件短而硬拆。

### 2. Chat runtime 仍然很重

当前普通聊天相关逻辑还集中在 `__init__.py`：

```text
图片缓存和图片上下文处理。
run_vision_graph。
run_memory_context_graph。
build_chat_prompt_context。
describe_chat_images。
run_memory_persist_graph。
run_chat_graph_session_runtime。
run_chat_via_root_graph。
handle_chat。
RootGraph/CHAT 观测格式化。
Shadow chat 快照辅助。
```

如果继续拆，建议优先考虑：

```text
chat_runtime_factory.py
  组装 ChatGraph / MemoryContextGraph / VisionGraph / MemoryPersistGraph 所需依赖。

chat_observation_runtime.py
  RootGraph/CHAT 最近观测和格式化。

image_context_runtime.py
  图片缓存、图片来源解析、图片 follow-up 文本合并。
```

但这块风险比 MainAgent owner 更高，因为它承载普通聊天主路径。建议只在 P2.5 回归稳定后再拆。

### 3. Diagnostics / MemoryRAG / MemoryAdmin 可继续 service 化

这些函数仍在 `__init__.py` 中：

```text
run_diagnostics_graph。
run_memory_retrieval_graph。
run_memory_admin_graph。
memory_rag_status_snapshot / memory_rag_status_lines。
summary_status_lines。
manual_long_term_context。
format_current_long_term_memories。
agent_ops_health_reply。
agent_vision_troubleshoot_reply。
agent_memory_rag_troubleshoot_reply。
```

它们已经被 `OwnerRuntimeFactory` 作为依赖注入给 owner runtime，所以后续可以独立拆：

```text
diagnostics_runtime.py
  DiagnosticsGraph 装配、状态页、最近错误、TTS health。

memory_rag_runtime.py
  MemoryRAG 状态、检索、重建、错误格式化。

memory_admin_runtime.py
  摘要、长期记忆、压缩、清理等管理图装配。
```

建议优先级：

```text
1. diagnostics_runtime.py
2. memory_rag_runtime.py
3. memory_admin_runtime.py
```

原因是 Diagnostics 多数只读，拆分风险最低；MemoryAdmin 包含写操作，应该放在后面。

### 4. Voice / Notification 可独立拆，但不是当前瓶颈

当前仍在 `__init__.py`：

```text
run_notification_graph。
run_voice_graph_intent。
send_tts_record。
handle_direct_or_last_tts。
tts_status_lines。
tts_status_reply_lines。
```

这块适合后续拆成：

```text
notification_runtime.py
voice_runtime.py
```

但它们和 MainAgent Runtime service / Web Owner Console 铺路关系较弱，可以晚一点处理。

## 后续拆分建议

### 推荐路线 A：先做 Web Owner Console 读模型设计

现在 MainAgent owner runtime 的 task/read/write/service 边界已经够清楚，可以先不继续拆 `__init__.py`，转向 Web Owner Console 的只读读模型设计。

建议范围：

```text
只设计，不接 HTTP。
先定义 Owner Console 需要哪些 read model。
复用 owner_agent_runtime 的任务工作台、任务详情、审批详情。
复用 owner_read_runtime 的只读命令结果。
不做写操作 UI。
不做审批确认按钮。
```

优点：

```text
能验证本轮解耦是否真的对未来入口有用。
不会继续影响普通聊天主路径。
更贴近 v1.6 Runtime service 铺路目标。
```

### 推荐路线 B：继续拆 Diagnostics runtime

如果继续代码拆分，建议从 Diagnostics 开始：

```text
新增 diagnostics_runtime.py。
迁移 run_diagnostics_graph、diagnostics_graph_runtime_lines、recent_errors_reply、tts_status_reply_lines。
保持 QQ 命令 /诊断、/配置状态、/视觉状态、/最近错误、/图片缓存状态 行为不变。
保持 owner_read_runtime 通过 factory 注入 diagnostics runner。
```

边界：

```text
不改诊断输出内容。
不改 DiagnosticsGraph node sequence。
不新增运行时副作用。
不改变 QQ 命令别名。
```

### 暂不推荐路线 C：继续拆 Chat runtime

普通聊天主路径还涉及：

```text
图片识别
MemoryRAG 注入
短期历史
摘要压缩
TTS 候选
RootGraph commit
Shadow snapshot
```

这一块继续拆当然有价值，但风险更大。建议等 P2.5 live 回归稳定后，再拆 Chat runtime。

## P2.5 建议完成标准

P2.5 不追求新增能力，完成标准是：

```text
1. 文档记录当前解耦状态和剩余职责。
2. 全量测试通过。
3. QQ live 验证 /agent task/read/write 关键路径。
4. 明确下一步是 Web Owner Console 读模型设计，还是继续拆 Diagnostics runtime。
5. 不引入新的运行时行为变化。
```

## 当前建议

当前建议优先选择：

```text
P2.6：Web Owner Console read-model 设计，不写前端、不接 HTTP。
```

理由：

```text
MainAgent owner runtime 已经具备复用边界。
先设计 read model 可以验证 task/read/write service 是否足够稳定。
风险小于继续拆普通聊天主路径。
能自然承接后续 Owner Console。
```

如果想继续代码拆分，则选择：

```text
P2.6-alt：Diagnostics runtime service 解耦。
```

但暂不建议立刻拆 Chat runtime。
