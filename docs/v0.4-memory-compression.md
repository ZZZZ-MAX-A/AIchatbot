# v0.4 记忆压缩方案

## 目标

v0.4 解决“消息越聊越多”带来的三个问题：

1. SQLite 原始消息不断增长。
2. 短期上下文里旧消息过多会干扰当前回复。
3. 后续长期记忆提取时，不能把大量闲聊、重复话和无效内容全部带进去。

压缩方案不替代 v0.3 的短期记忆，而是在它之上增加一层“会话摘要”。

## 记忆层级

当前推荐把记忆拆成四层：

```text
第一层：短期原文
第二层：会话摘要
第三层：长期记忆
第四层：语义索引
```

第一层：短期原文

```text
用途：接住最近对话
数据：最近若干条 user / assistant 原始消息
状态：v0.3 已实现
```

第二层：会话摘要

```text
用途：压缩较旧的一段聊天，保留这段聊天的主要内容
数据：某个会话中一段旧消息的摘要
状态：v0.4 已实现第一批
```

第三层：长期记忆

```text
用途：保存当前对话对象值得长期保留的大致摘要
数据：归档后的结构化记忆
状态：v0.3 已预留表结构
```

第四层：语义索引

```text
用途：从摘要和长期记忆里按语义召回相关旧信息
数据：embedding 向量
状态：v0.3 已预留长期记忆向量表，摘要向量可后续补充
```

## 配置建议

新增配置：

```env
MAX_CONTEXT_MESSAGES=20
MAX_STORED_MESSAGES_PER_SESSION=200
SUMMARY_KEEP_RECENT_MESSAGES=80
SUMMARY_BATCH_MESSAGES=120
MAX_SESSION_SUMMARIES_IN_CONTEXT=3
ENABLE_MEMORY_COMPRESSION=true
```

字段说明：

- `MAX_CONTEXT_MESSAGES`: 每次发给 AI 的最近原文消息数，当前已有。
- `MAX_STORED_MESSAGES_PER_SESSION`: 每个会话最多保留多少条原始消息。
- `SUMMARY_KEEP_RECENT_MESSAGES`: 压缩时至少保留最近多少条原文不动。
- `SUMMARY_BATCH_MESSAGES`: 每次压缩多少条旧消息。
- `MAX_SESSION_SUMMARIES_IN_CONTEXT`: 每次最多带入多少条会话摘要。
- `ENABLE_MEMORY_COMPRESSION`: 是否启用自动压缩。

推荐初始值：

```text
每次回复只带最近 20 条原文
每个会话最多保留 200 条原文
压缩时保留最近 80 条原文
每次把较旧的 120 条压缩成 1 条摘要
每次最多带入最近 3 条摘要
```

## 数据表

新增 `session_summaries` 表：

```sql
CREATE TABLE session_summaries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_key TEXT NOT NULL,
    message_type TEXT NOT NULL,
    user_id TEXT,
    group_id TEXT,
    summary TEXT NOT NULL,
    message_start_id INTEGER NOT NULL,
    message_end_id INTEGER NOT NULL,
    source_message_count INTEGER NOT NULL,
    created_at TEXT NOT NULL
);
```

字段说明：

- `session_key`: 会话标识，私聊为 `private:QQ号`，群聊为 `group:群号`。
- `message_type`: `private` 或 `group`。
- `user_id`: 私聊用户 QQ 号，群聊摘要可为空。
- `group_id`: 群聊群号，私聊摘要为空。
- `summary`: 这段旧聊天的摘要。
- `message_start_id`: 被摘要的第一条消息 ID。
- `message_end_id`: 被摘要的最后一条消息 ID。
- `source_message_count`: 本次摘要覆盖了多少条原始消息。
- `created_at`: 摘要生成时间。

后续如果给摘要也做语义索引，可以再加：

```sql
CREATE TABLE summary_embeddings (
    summary_id INTEGER PRIMARY KEY,
    provider TEXT NOT NULL,
    model TEXT NOT NULL,
    embedding_json TEXT NOT NULL,
    created_at TEXT NOT NULL,
    FOREIGN KEY(summary_id) REFERENCES session_summaries(id)
        ON DELETE CASCADE
);
```

v0.4 可以先不实现 `summary_embeddings`，只预留设计。

## 压缩触发规则

每次 AI 成功回复并写入 `messages` 后，检查当前会话的原始消息数量。

当满足以下条件时触发压缩：

```text
当前会话 messages 数量 > MAX_STORED_MESSAGES_PER_SESSION
并且 ENABLE_MEMORY_COMPRESSION=true
```

压缩范围：

```text
只压缩当前会话最旧的一批消息
不压缩最近 SUMMARY_KEEP_RECENT_MESSAGES 条消息
每次最多压缩 SUMMARY_BATCH_MESSAGES 条消息
```

例子：

```text
当前会话有 230 条原文
MAX_STORED_MESSAGES_PER_SESSION=200
SUMMARY_KEEP_RECENT_MESSAGES=80
SUMMARY_BATCH_MESSAGES=120
```

处理方式：

```text
取最旧的 120 条消息
生成 1 条摘要
写入 session_summaries
删除这 120 条原文
保留最近 110 条原文
```

这样数据库不会无限积累原始聊天，但又能保留旧对话的大意。

## 摘要生成要求

摘要必须控制长度，不能把旧消息逐句复述。

摘要内容必须保持客观，不把没有发生的事写进去，也不替用户补充动机。

当前 v0.4 不让摘要承担“像人一样说话”的任务。摘要只负责压缩聊天记录，保留对后续对话和记忆归档有用的信息。

建议系统提示词：

```text
你是聊天记录压缩器。请把下面一段 QQ 聊天压缩成简洁摘要，只保留对后续对话有帮助的信息。
不要编造，不要记录无意义寒暄，不要保存隐私敏感内容。
重点保留：已确认的事实、正在进行的任务、未完成事项、重要决定、群聊主题。
输出 200 字以内中文摘要。
```

摘要内容建议包含：

```text
这段对话主要讨论了什么
已经确定的方案或决定
后续还要继续做的事情
对方明显稳定的表达习惯
```

摘要内容不建议包含：

```text
无意义寒暄
重复催促
临时情绪
API Key、Token、二维码链接、密码
过长原文复述
把机器人或用户没有表达过的想法写成事实
```

## 上下文组装

v0.4 后，调用 AI 时建议按这个顺序组装：

```text
系统提示词
  + 当前会话最近的会话摘要
  + 当前用户/当前群的长期记忆
  + 最近 MAX_CONTEXT_MESSAGES 条短期原文
  + 用户本次消息
```

其中会话摘要可以作为一条 system 或 assistant 上下文，例如：

```text
以下是当前会话较早内容的摘要：
1. ...
2. ...
```

摘要不应该太多，初始建议最多带入最近 3 条。

## QQ 命令

建议新增主人命令：

```text
/摘要状态
/查看摘要
/压缩当前会话
/清空当前摘要
/清空全部摘要
```

命令说明：

- `/摘要状态`: 查看摘要数量、覆盖消息数量、当前压缩配置。
- `/查看摘要`: 查看当前会话最近几条摘要。
- `/压缩当前会话`: 主人手动触发当前会话压缩。
- `/清空当前摘要`: 清空当前私聊或当前群聊的摘要。
- `/清空全部摘要`: 清空全部会话摘要。

普通用户不允许执行这些命令。

## 与长期记忆的边界

会话摘要和长期记忆不能混在一起。

会话摘要回答的是：

```text
这段聊天之前说了什么？
```

长期记忆回答的是：

```text
这个人、这个群、这个机器人有什么稳定特点？
```

例子：

```text
会话摘要：用户和机器人讨论了 v0.3 SQLite 记忆方案，并测试通过短期记忆。
长期回忆摘要：当前对话对象已经完成权限、短期记忆和摘要压缩测试，下一步关注长期回忆摘要。
```

摘要可以作为长期记忆提取的来源，但不能直接等同于长期记忆。

## 隐私规则

压缩不会让数据离开模型调用链，但摘要仍然可能包含真实聊天内容，因此继续存放在：

```text
data/chatbot.db
```

不要提交数据库文件。

摘要生成时必须避免保存：

```text
API Key
Token
二维码链接
账号密码
身份证、手机号等敏感个人信息
```

## 实现状态

已实现：

1. 压缩配置。
2. `session_summaries` 表。
3. 摘要读写、统计和清空。
4. 自动压缩。
5. 手动压缩当前会话。
6. 摘要参与上下文组装。
7. 主人摘要管理命令。

暂未实现：

1. 摘要语义索引。
2. 长期记忆自动提取。
3. 摘要和长期记忆的自动合并。

## 实现步骤

当前第一批按以下顺序实现：

1. 在 `.env.example` 和 `config/.env.example` 增加压缩配置。
2. 在 `database.py` 增加 `session_summaries` 表。
3. 新增 `summaries.py`，负责摘要读写、统计、清空。
4. 新增 `compressor.py`，负责判断是否需要压缩和调用 AI 生成摘要。
5. 修改 `memory.py`，读取上下文时可带入最近摘要。
6. 修改 `__init__.py`，AI 成功回复后尝试自动压缩。
7. 增加主人命令：`/摘要状态`、`/查看摘要`、`/压缩当前会话`、`/清空当前摘要`、`/清空全部摘要`。
8. 更新 `README.md` 和 `docs/runbook.md`。

## 测试计划

基础测试：

1. 正常聊天，确认短期记忆不受影响。
2. 人工插入超过阈值的测试消息，确认可以生成摘要。
3. 压缩后确认旧原文被删除，最近原文被保留。
4. `/查看摘要` 能看到当前会话摘要。
5. 重启机器人后摘要仍然存在。
6. AI 回复能利用摘要里的旧信息。
7. `/清空当前摘要` 只清空当前会话摘要。
8. `/清空全部摘要` 清空所有摘要。
9. 非主人执行摘要管理命令会被拒绝。

回归测试：

1. `/记忆状态` 正常。
2. `/重置` 仍然只清空当前短期原文。
3. `/清空全部上下文` 仍然只清空短期原文，不清空摘要。
4. 权限、白名单、黑名单逻辑不受影响。
